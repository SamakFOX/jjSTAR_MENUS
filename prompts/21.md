# Drag & Drop 방식을 아이템 기준이 아닌 Gap 기준으로 변경 요청

## 현재 문제

현재 드래그/드롭 위치 계산이 아이템 기준으로 되어 있어 문제가 발생합니다.

예시:

- 특정 아이템 위에 드래그
- preview는 아이템 앞에 들어갈 것처럼 보임
- 실제 drop 후에는 아이템 뒤에 들어감
- before / after 판정이 계속 헷갈림
- 그리드형 Visual Mode에서는 X축/Y축 계산이 복잡해져 위치가 어긋남

현재 방식은 `targetItem` 위에 마우스를 올린 뒤  
그 아이템의 앞인지 뒤인지 계산하는 구조로 보입니다.

하지만 이 방식은 Visual Mode처럼 카드가 그리드로 배치된 화면에서는 불안정합니다.

---

# 수정 방향

드롭 위치를 **아이템 위**가 아니라 **아이템과 아이템 사이의 gap/drop slot** 기준으로 처리해주세요.

즉, 사용자가 드래그할 때는 아래처럼 동작해야 합니다.

```txt
[아이템 A]   [아이템 B]   [아이템 C]

아이템 B를 드래그해서 아이템 A와 C 사이로 옮길 때:

[아이템 A]   [드롭 슬롯]   [아이템 C]
````

이때 drop target은 아이템 A 또는 아이템 C가 아니라
**아이템 사이에 생긴 drop slot**입니다.

---

# 핵심 개념

기존 방식:

```txt
targetItem 위에 올림
→ before / after 계산
→ insertIndex 계산
```

변경 방식:

```txt
아이템 사이 gap에 올림
→ 해당 gap의 insertIndex를 직접 사용
→ before / after 계산 불필요
```

즉, Visual Mode에서는 가능하면 `before`, `after` 판정을 줄이고
**insertIndex 자체를 drop slot이 가지도록** 해주세요.

---

# Drop Slot 구조 예시

Visual Mode에서 렌더링할 때 각 아이템 사이에 보이지 않는 drop slot을 둡니다.

```txt
slot-0
[Item 0]
slot-1
[Item 1]
slot-2
[Item 2]
slot-3
```

드래그 중에는 해당 slot이 hover되면 양쪽 아이템이 벌어지고,
그 위치가 실제 insertIndex가 됩니다.

```js
const dropSlot = {
  parentId,
  level,
  insertIndex,
};
```

---

# Drop Intent 예시

아이템 기준이 아니라 slot 기준으로 dropIntent를 만들어주세요.

```js
const dropIntent = {
  dropMode: 'reorder',
  dragId,
  level,
  newParentId,
  insertIndex,
};
```

중요한 점:

```txt
targetItemId가 필수값이 아니어도 됩니다.
핵심은 insertIndex입니다.
```

기존처럼 targetItem 기준으로 `before` / `after`를 계산하지 말고,
slot이 가진 `insertIndex`를 그대로 적용해주세요.

# DropSlot hover 처리

```js
function handleDropSlotHover(slot) {
  const intent = {
    dropMode: 'reorder',
    dragId: draggingItem.id,
    level: slot.level,
    newParentId: slot.parentId,
    insertIndex: slot.insertIndex,
  };

  setDropIntent(intent);
}
```

---

# 실제 drop 처리

```js
function applyDropToSlot(intent, nodes) {
  const movingGroup = getMovingGroup(intent.dragId, nodes);
  const movingIds = new Set(movingGroup.map((node) => node.id));

  const remainingNodes = nodes.filter((node) => !movingIds.has(node.id));

  const updatedGroup = movingGroup.map((node) => {
    if (node.id === intent.dragId) {
      return {
        ...node,
        level: intent.level,
        parentId: intent.newParentId,
      };
    }

    return node;
  });

  return [
    ...remainingNodes.slice(0, intent.insertIndex),
    ...updatedGroup,
    ...remainingNodes.slice(intent.insertIndex),
  ];
}
```

단, `insertIndex`가 전체 nodes 기준인지, 같은 parent 그룹 기준인지 반드시 통일해주세요.

# 권장 방식

Visual Mode는 전체 flat nodes 배열 기준으로 직접 insertIndex를 계산하기보다,
**부모별 그룹 단위로 slot을 계산**하는 방식이 더 안전합니다.

예시:

```txt
Level 1 그룹
- Level 2 slot 0
- Level 2 item A
- Level 2 slot 1
- Level 2 item B
- Level 2 slot 2

Level 2 그룹
- Level 3 slot 0
- Level 3 item A
- Level 3 slot 1
- Level 3 item B
- Level 3 slot 2
```

이렇게 하면 Level 2는 Level 1 안의 slot에만 들어가고,
Level 3은 Level 2 안의 slot에만 들어갈 수 있습니다.

---

# Level별 Drop Slot 허용 규칙

## Level 1

Level 1은 최상위 slot에만 들어갈 수 있습니다.

```txt
Level 1 reorder slot만 허용
parentId 없음
```

---

## Level 2

Level 2는 Level 1 내부의 Level 2 slot에만 들어갈 수 있습니다.

허용:

```txt
Level 1 내부 Level 2 slot
```

금지:

```txt
Level 2 내부 slot
Level 3 주변 slot
Level 3 하위 slot
```

즉, Level 2를 드래그 중일 때는
Level 3 근처의 slot은 표시하지 않거나 비활성화해주세요.

---

## Level 3

Level 3은 Level 2 내부의 Level 3 slot에만 들어갈 수 있습니다.

허용:

```txt
Level 2 내부 Level 3 slot
```

금지:

```txt
Level 1 내부 slot
Level 2와 같은 레벨 slot
Level 3 하위 slot
```

# UI 동작

드래그 중에는 아이템 위에 노란 박스를 표시하는 대신,
실제 들어갈 위치에 gap을 만들어주세요.

예시:

```txt
[대학생활 상담]   [학생증]

학생회 활동을 둘 사이에 넣으려는 경우:

[대학생활 상담]   [  학생회 활동 들어갈 자리  ]   [학생증]
```

이렇게 하면 사용자는 “어느 아이템 앞/뒤인지”를 추측하지 않아도 됩니다.

---

# Visual Mode에서 필요한 표시

드래그 중 drop 가능한 slot은 아래처럼 표시해주세요.

```txt
- 양쪽 아이템이 벌어짐
- 사이에 얇은 노란색 라인 또는 박스 표시
- 가능하면 드래그 중인 아이템의 placeholder 크기만큼 공간 확보
```

금지:

```txt
- target item 전체를 감싸서 before/after처럼 보이게 하기
- target item 위 hover만으로 drop 위치 결정하기
- preview와 실제 drop 위치를 다르게 계산하기
```

---

# 핵심 요구사항

현재 문제는 before/after 계산이 불안정한 것입니다.

따라서 Visual Mode에서는 아래 방식으로 바꿔주세요.

```txt
아이템 위에 drop하는 방식 X
아이템 사이 slot에 drop하는 방식 O
```

그리고 drop slot이 직접 `insertIndex`, `newParentId`, `level`을 가져야 합니다.

```js
const dropSlot = {
  level,
  parentId,
  insertIndex,
};
```

실제 drop 시에는 이 값을 그대로 사용합니다.

# 완료 기준

| 테스트                | 기대 결과                             |
| ------------------ | --------------------------------- |
| 아이템 사이로 드래그        | 양쪽 아이템이 벌어짐                       |
| 벌어진 사이에 drop       | 정확히 그 사이에 삽입                      |
| 기다렸다 drop          | 위치 일치                             |
| 바로 drop            | 위치 일치                             |
| before/after 계산 오류 | 발생하지 않음                           |
| 오른쪽으로 +1 밀림        | 발생하지 않음                           |
| 왼쪽으로 -1 밀림         | 발생하지 않음                           |
| Level 2 드래그        | Level 1 내부 Level 2 slot에만 drop 가능 |
| Level 3 드래그        | Level 2 내부 Level 3 slot에만 drop 가능 |
| Level 4 생성         | 절대 없음                             |

# 최종 요약

드래그/드롭 위치를 더 이상 target item 기준으로 계산하지 말고,
**아이템과 아이템 사이에 생성되는 drop slot 기준으로 계산해주세요.**

핵심은 아래입니다.

```txt
양쪽 아이템이 밀려난 후 생긴 그 사이 위치 = 실제 drop 위치
```

따라서 dropIntent는 targetItemId보다
`insertIndex`, `parentId`, `level` 중심으로 구성해주세요.

```js
const dropIntent = {
  dragId,
  level,
  newParentId,
  insertIndex,
};
```

이 방식이면 before / after 판정 오류나
오른쪽으로 +1 밀리는 문제가 크게 줄어듭니다.

```
