# 사용자 가이드 개선 요청: 인터랙티브 튜토리얼 방식으로 변경

## 작업 목표

현재 사용자 가이드는 기본적으로 동작하지만, 실제 사용성 측면에서 아래 문제가 있습니다.

## 현재 문제

```txt
1. 설명창이 강조 영역 뒤에 가려져 일부 내용이 보이지 않음
2. 마우스 포인터/드래그 애니메이션이 하이라이트 영역 뒤에 깔려 보이지 않음
3. 버튼이 [<] [다음] 형태라 의미가 불명확함
4. 처음 진입 가이드가 끝난 뒤 편집 화면 내부 가이드로 이어지지 않음
5. “사용 방법 보기” 버튼이 편집 화면에만 있어서 첫 화면에서 다시 보기 어려움
6. 일부 가이드 레이어가 기존 가이드 레이어 뒤에 깔려 보이지 않음
7. 툴팁 설명은 실제 hover된 것처럼 보이지 않아 이해가 약함
8. 설명창 위치가 애매해서 대상 요소와 연결성이 떨어짐
9. 각 화면에 들어가야만 가이드가 따로 나와서 전체 흐름을 익히기 어려움
10. 사용자가 직접 조작하지 않아도 다음으로 넘어가므로 대충 넘길 가능성이 큼
````

따라서 가이드를 단순 설명 팝업이 아니라
**처음부터 끝까지 자동으로 화면을 전환하며 안내하고, 중요한 단계는 사용자가 직접 조작해야 다음으로 넘어가는 튜토리얼 방식**으로 개선해 주세요.

---

# 1. 가장 중요한 조건

기존에 정상 동작 중인 기능은 절대 건드리지 말아 주세요.

```txt
로그인/인증 로직
메뉴 편집 로직
리스트모드 드래그앤드랍
비주얼모드 드래그앤드랍
임시저장/자동저장
최종 제출 저장
변경 로그 저장
관리자 대시보드
미리보기
툴팁 기능
undo/redo
초기화
Supabase 저장 구조
```

이번 작업은 **가이드 UX 개선만** 대상으로 합니다.

가이드 때문에 실제 데이터가 변경되면 안 됩니다.

```txt
가이드 표시만으로 menuState 변경 금지
가이드 표시만으로 commitMenuChange 호출 금지
가이드 표시만으로 undo/redo 증가 금지
가이드 표시만으로 change_log 추가 금지
가이드 표시만으로 임시저장/자동저장 실행 금지
```

# 2. 가이드 레이어 z-index 문제 수정

## 현재 문제

설명창, 포인터 아이콘, 애니메이션 요소가 하이라이트 박스나 기존 UI 뒤에 깔려 보이지 않는 문제가 있습니다.

특히 마우스 포인터 아이콘이 하이라이트 영역 뒤에 있어서 애니메이션이 잘 보이지 않습니다.

## 수정 요청

가이드 관련 레이어의 z-index 체계를 정리해 주세요.

권장 순서:

```txt
기본 화면 UI: z-index 0~100
반투명 dim overlay: z-index 9000
강조 대상 clone/highlight: z-index 9010
가이드 포인터/손가락 애니메이션: z-index 9020
가이드 설명창: z-index 9030
가이드 버튼/닫기: z-index 9040
```

예시:

```css
.guide-dim {
  z-index: 9000;
}

.guide-highlight {
  z-index: 9010;
}

.guide-pointer {
  z-index: 9020;
}

.guide-popover {
  z-index: 9030;
}

.guide-controls {
  z-index: 9040;
}
```

## 중요

```txt
포인터 애니메이션은 항상 하이라이트보다 위에 보여야 함
설명창은 항상 하이라이트보다 위에 보여야 함
설명창이 기존 UI나 다른 가이드 레이어 뒤에 숨으면 안 됨
```

# 3. 설명창이 대상 요소를 가리지 않도록 배치

## 현재 문제

설명창이 강조 영역을 덮어서 실제 설명 대상이 보이지 않거나 클릭이 막히는 문제가 있습니다.

## 수정 요청

설명창은 대상 요소 바로 위나 아래에 배치하되,
대상 요소를 가리지 않는 위치로 자동 계산해 주세요.

## 배치 우선순위

```txt
1순위: 대상 요소 아래
2순위: 대상 요소 위
3순위: 대상 요소 오른쪽
4순위: 대상 요소 왼쪽
```

화면 밖으로 나가면 자동으로 다른 위치를 선택해 주세요.

## 예시

```js
function getGuidePopoverPlacement(targetRect, popoverSize, viewportSize) {
  if (targetRect.bottom + popoverSize.height + 16 < viewportSize.height) {
    return "bottom";
  }

  if (targetRect.top - popoverSize.height - 16 > 0) {
    return "top";
  }

  if (targetRect.right + popoverSize.width + 16 < viewportSize.width) {
    return "right";
  }

  return "left";
}
```

## 기대 결과

```txt
설명창이 강조 대상 위를 덮지 않음
설명창과 대상 요소의 관계가 명확함
설명창이 화면 밖으로 잘리지 않음
작은 화면에서도 내용이 보임
```

# 4. [<] [다음] 버튼을 [이전] [다음]으로 변경

## 현재 문제

현재 가이드 버튼이 `[<] [다음]`처럼 표시되어 있어 의미가 약합니다.

## 수정 요청

명확하게 텍스트 버튼으로 표시해 주세요.

```txt
[이전] [다음]
```

마지막 단계에서는:

```txt
[이전] [완료]
```

첫 단계에서는 이전 버튼을 비활성화하거나 숨겨도 됩니다.

```txt
첫 단계: [건너뛰기] [다음]
중간 단계: [이전] [다음]
마지막 단계: [이전] [완료]
```

## 다시 보지 않기

`다시 보지 않기`는 하단 좌측에 유지해 주세요.

# 5. 첫 화면부터 전체 튜토리얼이 이어지게 변경

## 현재 문제

처음 진입 시 가이드가 2개만 나오고, `알겠어요`를 누르면 가이드가 끝납니다.
그 후 사용자가 `메뉴 편집하기`를 눌러도 편집 화면 내부 가이드가 이어지지 않습니다.

이러면 전체 사용 흐름을 익히기 어렵습니다.

## 수정 요청

가이드는 **처음 화면 → 편집 화면 → 비주얼모드 → 최종 제출 화면**까지 하나의 흐름으로 이어져야 합니다.

즉, 사용자가 처음 가이드를 시작하면 페이지/상태가 자동으로 전환되면서 끝까지 안내해 주세요.

# 6. 전체 가이드 플로우 제안

## 전체 단계

```txt
Step 1. 첫 화면: 메뉴 편집하기 버튼 안내
Step 2. 첫 화면: 상단 버튼 기능 안내
Step 3. 편집 화면 진입: 리스트모드 구조 안내
Step 4. 편집 화면: 드래그 핸들 안내
Step 5. 편집 화면: 아이템별 편집 버튼 안내
Step 6. 편집 화면: 메뉴 설명 hover 툴팁 안내
Step 7. 편집 화면: 임시저장/미리보기/최종제출 안내
Step 8. 비주얼모드 전환: 비주얼모드 편집 방법 안내
Step 9. 최종 제출 화면 이동: 제출 의도 작성 안내
Step 10. 최종 제출 화면: 취소/제출 버튼 안내
Step 11. 완료
```

# 7. 자동 화면 전환형 가이드

## 요구사항

가이드가 특정 단계에 도달하면 필요한 화면으로 자동 전환되어야 합니다.

예시:

```txt
Step 1에서 사용자가 "메뉴 편집하기"를 실제로 누름
→ 편집모드로 전환
→ Step 3 자동 시작

Step 8에서 "화면에서 편집" 탭 안내
→ 사용자가 직접 클릭
→ 비주얼모드로 전환
→ 비주얼모드 가이드 시작

Step 9에서 "최종 제출" 안내
→ 사용자가 직접 클릭
→ 최종 제출 확인 화면으로 이동
→ 제출 화면 가이드 시작
```

## 중요

이 전환은 가이드용이지만 실제 사용자가 누른 버튼으로 진행되어야 합니다.

다만 최종 제출 단계에서는 실제 제출까지는 하지 않아야 합니다.
최종 제출 화면까지만 이동해서 설명하고, 제출 버튼은 안내만 해 주세요.

# 8. 사용자가 직접 조작해야 다음으로 넘어가는 방식

## 현재 문제

현재는 사용자가 아무 조작을 하지 않아도 `다음`만 누르면 넘어갑니다.
그러면 안내를 읽지 않고 넘길 가능성이 큽니다.

## 수정 요청

중요한 단계는 사용자가 실제로 조작해야 다음 단계로 넘어가게 해 주세요.

## 인터랙션 체크가 필요한 단계

### Step 1. 메뉴 편집하기

```txt
목표: 사용자가 "메뉴 편집하기" 버튼 클릭
완료 조건: editMode 진입
```

### Step 4. 드래그 핸들 확인

```txt
목표: 사용자가 드래그 핸들에 마우스를 올리거나 클릭/드래그 시작
완료 조건: 핸들 hover 또는 pointerdown 감지
```

실제 메뉴 위치를 바꾸지 않아도 됩니다.
처음에는 `핸들을 눌러보세요` 정도로 충분합니다.

### Step 5. 아이템별 편집 버튼 확인

```txt
목표: 사용자가 수정/추가/삭제 버튼 영역에 마우스 hover
완료 조건: 버튼 영역 hover 감지
```

### Step 6. 툴팁 확인

```txt
목표: 사용자가 메뉴명에 마우스를 올림
완료 조건: desc tooltip 표시 감지
```

### Step 8. 비주얼모드 전환

```txt
목표: 사용자가 "화면에서 편집" 탭 클릭
완료 조건: visualMode 활성화
```

### Step 9. 최종 제출 화면

```txt
목표: 사용자가 "최종 제출" 버튼 클릭
완료 조건: submit confirm 화면 진입
```

### Step 10. 제출 의도 입력

```txt
목표: textarea에 한 글자 이상 입력해보기
완료 조건: intention textarea input 감지
```

단, 가이드 중 실제 제출까지 강제하지는 않습니다.

# 9. 조작형 단계의 버튼 처리

조작이 필요한 단계에서는 `다음` 버튼을 비활성화하거나 안내 문구를 표시해 주세요.

예시:

```txt
먼저 강조된 핸들을 눌러보세요.
```

조작이 감지되면:

```txt
좋아요! 이제 다음 단계로 넘어갈 수 있어요.
```

그리고 `다음` 버튼 활성화.

# 10. 드래그 핸들 애니메이션 개선

## 현재 문제

드래그 가이드 애니메이션이 아이템 전체를 움직이는 것처럼 보여서 실제 사용 방식과 다릅니다.

## 수정 요청

드래그 애니메이션은 반드시 **핸들 위치**에서 시작해야 합니다.

## 원하는 애니메이션

```txt
1. 핸들 영역 pulse 강조
2. 포인터 아이콘이 핸들 위로 이동
3. 포인터가 핸들을 누르는 모션
4. 핸들에서 아래/위로 살짝 드래그하는 모션
5. 실제 아이템은 이동하지 않거나 ghost preview만 아주 약하게 표시
```

## 중요

```txt
아이템 전체가 실제로 이동하는 것처럼 보이면 안 됨
포인터는 하이라이트보다 위에 있어야 함
애니메이션은 실제 데이터 변경 없이 CSS transform으로만 처리
```

# 11. 툴팁 가이드 개선

## 현재 요청

툴팁 기능 설명은 단순히 말로 설명하는 것보다,
마우스 포인터가 메뉴명 위로 이동해서 실제 hover된 것처럼 툴팁이 뜨는 모습을 보여주면 좋겠습니다.

## 수정 요청

툴팁 가이드 단계에서는 아래처럼 보여주세요.

```txt
1. 포인터 아이콘이 특정 메뉴명으로 이동
2. 메뉴명 hover 상태처럼 표시
3. desc 툴팁 예시가 표시됨
4. 설명창에는 "메뉴명이 헷갈리면 마우스를 올려 설명을 확인할 수 있어요" 표시
```

## 중요

기존 desc tooltip 기능을 실제로 사용해도 되고,
가이드 전용 예시 툴팁을 띄워도 됩니다.

단, 가이드 전용 예시 툴팁은 실제 데이터에 영향을 주면 안 됩니다.

# 12. “사용 방법 보기” 버튼 위치 변경

## 현재 문제

`사용 방법 보기` 버튼이 편집 화면 내부에만 있어,
첫 화면에서는 가이드를 다시 시작하기 어렵습니다.

## 수정 요청

`사용 방법 보기` 버튼은 첫 화면부터 보이게 해 주세요.

## 표시 위치

첫 화면 상단 버튼 영역 또는 안내 박스에 표시:

```txt
사용 방법 보기
```

권장 위치:

```txt
[메뉴 편집하기] 버튼 옆
또는 우측 상단 미리보기/최종제출 영역 근처
```

## 동작

클릭 시 항상 전체 튜토리얼을 처음부터 시작합니다.

```txt
사용 방법 보기 클릭
→ Step 1부터 전체 가이드 시작
```

# 14. target 등록 방식

가이드 대상 요소를 안정적으로 찾기 위해 `data-guide-id`를 부여해 주세요.

예시:

```jsx
<button data-guide-id="edit-button">
  메뉴 편집하기
</button>
```

```jsx
<div data-guide-id="top-actions">
  ...
</div>
```

```jsx
<div data-guide-id="drag-handle">
  ...
</div>
```

```jsx
<div data-guide-id="item-actions">
  ...
</div>
```

```jsx
<button data-guide-id="visual-mode-tab">
  화면에서 편집
</button>
```

```jsx
<button data-guide-id="final-submit-button">
  최종 제출
</button>
```

이렇게 하면 화면이 바뀌어도 가이드가 target을 찾기 쉽습니다.

# 15. 가이드 팝업 위치 계산

각 단계마다 target rect를 가져와 팝업을 배치해 주세요.

```js
const targetEl = document.querySelector(`[data-guide-id="${target}"]`);
const rect = targetEl?.getBoundingClientRect();
```

팝업은 target을 가리지 않도록 자동 배치합니다.

```txt
bottom → top → right → left 순서로 가능한 위치 선택
```

# 16. 스크롤 처리

대상 요소가 화면 밖에 있으면 자동으로 스크롤해 주세요.

```js
targetEl.scrollIntoView({
  behavior: "smooth",
  block: "center",
  inline: "center",
});
```

스크롤 후 rect를 다시 계산해 주세요.

# 17. 첫 접속 / 다시 보지 않기

기존처럼 draft가 없는 사용자에게는 자동으로 가이드를 보여줄 수 있습니다.

```txt
draft 없음 → 전체 가이드 자동 시작
draft 있음 → 자동 시작 안 함
```

`다시 보지 않기`를 누르면 localStorage에 저장합니다.

```js
localStorage.setItem(`hideGuide:${authCode}`, "true");
```

단, `사용 방법 보기` 버튼을 누르면 다시 볼 수 있어야 합니다.

```txt
다시 보지 않기 상태여도 사용 방법 보기 클릭 시 가이드 시작
```

# 18. 가이드 도중 나가기

가이드에서 `건너뛰기` 또는 X 버튼을 누르면 즉시 종료합니다.

```txt
guideActive false
guideStep 초기화
화면 상태는 현재 상태 유지
실제 데이터 변경 없음
```

# 19. 최종 제출 화면 가이드 주의

최종 제출 화면의 가이드는 실제 제출까지 진행하면 안 됩니다.

## 허용

```txt
최종 제출 화면 진입
textarea 설명
취소/제출 버튼 설명
```

## 금지

```txt
가이드가 자동으로 제출 버튼 클릭
가이드가 실제 submission 생성
가이드가 intention 저장
```

사용자가 직접 최종 제출하는 것은 기존 기능대로 유지합니다.

# 20. 최종 기대 결과

## 개선 후 기대 동작

```txt
처음 들어온 사용자는 가이드가 시작됨
가이드는 첫 화면부터 편집 화면, 비주얼모드, 최종 제출 화면까지 이어짐
필요한 단계에서는 사용자가 직접 조작해야 다음으로 넘어감
가이드 팝업은 대상 요소를 가리지 않음
포인터 애니메이션은 하이라이트 위에 보임
드래그 안내는 아이템 전체가 아니라 핸들 중심으로 보임
툴팁 안내는 실제 hover처럼 보임
[이전] [다음] 버튼이 명확하게 표시됨
사용 방법 보기 버튼은 첫 화면부터 제공됨
기존 기능은 변경되지 않음
```

# 21. 핵심 요약

현재 가이드는 각 화면별 설명에 가까워서 사용자가 전체 흐름을 기억하기 어렵습니다.

따라서 아래 방식으로 개선해 주세요.

```txt
1. 첫 화면부터 끝까지 이어지는 전체 튜토리얼로 변경
2. 화면이 필요한 단계에서는 자동 또는 사용자 조작으로 전환
3. 중요한 단계는 직접 조작해야 다음으로 넘어가게 처리
4. 설명창은 대상 위/아래에 배치해 가리지 않게 처리
5. 포인터/애니메이션은 하이라이트보다 위에 표시
6. 드래그 안내는 핸들 위치 중심으로 수정
7. 툴팁 안내는 실제 hover 시연처럼 표시
8. 사용 방법 보기 버튼은 첫 화면부터 표시
9. 기존 기능은 절대 건드리지 않기
```
