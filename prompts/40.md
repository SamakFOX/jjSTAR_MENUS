# 최종 제출 흐름 개선 요청: 기본 팝업 제거 + 제출 확인/설문 화면 추가

## 작업 목표

현재 `최종 제출` 버튼을 누르면 브라우저 기본 confirm 팝업이 표시됩니다.

예시:

```txt
localhost:3000 내용:
현재 메뉴 배치를 최종 제출하시겠습니까?
[확인] [취소]
````

이 기본 팝업 대신, 네이버폼 설문지처럼 별도의 제출 확인 화면을 보여주고 싶습니다.

제출 전 마지막으로 간단한 의견을 입력받고, 사용자가 직접 제출 버튼을 눌러 최종 제출하도록 변경해 주세요.

---

## 가장 중요한 조건

현재 다른 기능들은 모두 잘 동작하고 있으므로, 이번 작업에서는 **최종 제출 흐름만 수정**해 주세요.

아래 기능은 절대 수정하지 말아 주세요.

```txt
로그인/인증 기능
리스트모드
비주얼모드
드래그앤드랍 로직
메뉴 이동/정렬 로직
되돌리기/다시실행
초기화
미리보기 기능
프로필 이미지
폰트 설정
우측 편집 도구
메뉴 데이터 구조
Supabase 저장 구조
```

이번 수정 범위는 오직 다음입니다.

```txt
최종 제출 버튼 클릭 이후의 화면/흐름
```

---

# 1. 기존 기본 confirm 팝업 제거

현재 최종 제출 시 아래와 같은 브라우저 기본 팝업이 뜹니다.

```js
confirm("현재 메뉴 배치를 최종 제출하시겠습니까?");
```

또는 유사한 confirm 로직이 있다면 제거해 주세요.

브라우저 기본 confirm/alert/prompt는 사용하지 말아 주세요.

```txt
confirm 사용 금지
alert 사용 금지
prompt 사용 금지
```

---

# 2. 최종 제출 버튼 클릭 시 제출 확인 화면으로 이동

상단의 `최종 제출` 버튼을 누르면 바로 서버 제출을 하지 말고,
먼저 제출 확인/설문 화면으로 이동하게 해 주세요.

## 기대 흐름

```txt
메뉴 편집 화면
→ 최종 제출 버튼 클릭
→ 제출 확인/설문 화면 표시
→ 사용자가 마지막 질문 작성/선택
→ 제출 버튼 클릭
→ 실제 최종 제출 처리
→ 제출 완료 화면 표시
```

---

# 3. 제출 확인/설문 화면 디자인 방향

첨부한 네이버폼 예시처럼 어두운 배경의 제출 확인 화면을 만들어 주세요.

화면 구성은 다음과 같습니다.

```txt
상단: PC 화면 / 모바일 화면 탭
카드: 이 상태로 제출하시겠습니까
질문 1: 메뉴 구성 의도 입력
질문 2: 최종 제출 확인 선택
하단: 제출 버튼
```

단, 완전히 네이버폼을 복제할 필요는 없고, 현재 프로젝트 디자인에 맞게 깔끔하게 구현하면 됩니다.

---

## 화면 상단 탭

상단에는 아래 탭을 표시해 주세요.

```txt
PC 화면
모바일 화면
```

현재는 PC 화면을 기본 활성화하면 됩니다.

모바일 화면은 추후 확장용으로 둘 수 있습니다.

```txt
PC 화면: 활성
모바일 화면: 비활성
```

모바일 화면 탭 클릭 기능은 당장 구현하지 않아도 됩니다.
필요하면 비활성 또는 준비중 상태로 처리해 주세요.

# 4. 제출 확인 카드

상단 카드에는 다음 문구를 표시해 주세요.

```txt
이 상태로 제출하시겠습니까
```

보조 문구는 아래처럼 넣어도 됩니다.

```txt
현재 구성한 메뉴 배치가 최종 제출됩니다.
```

또는:

```txt
제출 후에는 현재 메뉴 구조와 변경 로그가 저장됩니다.
```

# 5. 질문 1: 메뉴 구성 의도

첫 번째 질문은 주관식 입력입니다.

## 질문 제목

```txt
1. 메뉴 구성 의도가 어떻게 되나요?
```

## 예시 설명

질문 아래에는 예시 문구를 표시해 주세요.

```txt
ex) 학교 다니면서 저희가 써보기에 수강과 연관된 메뉴들은 모여있으면 좋을 것 같고 부가적인 학교 서비스도 각각 모여있으면 좋을 것 같아서 이렇게 구성했어요. 특히 나의메뉴는 정말 내 졸업을 위한 정보같은 나와 직접 연관되는 메뉴들로 구성해봤어요.
```

## 입력 방식

textarea로 답변을 입력받습니다.

placeholder:

```txt
답변을 입력해주세요.
```

## 필수 여부

이 질문은 필수로 해 주세요.

답변이 비어 있는 상태에서 제출 버튼을 누르면 제출하지 않고 안내 문구를 보여주세요.

예시:

```txt
메뉴 구성 의도를 입력해주세요.
```

# 6. 질문 2: 최종 제출 확인

두 번째 질문은 라디오 선택입니다.

## 질문 제목

```txt
2. 위 메뉴 배치로 최종 제출하시겠습니까?
```

## 선택지

```txt
네
취소
```

## 필수 여부

이 질문도 필수로 해 주세요.

선택하지 않은 상태에서 제출하면 안내 문구를 보여주세요.

예시:

```txt
최종 제출 여부를 선택해주세요.
```

---

## 선택값 처리

`네`를 선택하고 제출하면 실제 최종 제출을 진행합니다.

`취소`를 선택하고 제출하면 실제 제출하지 않고 편집 화면으로 돌아가게 해 주세요.

또는 취소 선택 시 제출 버튼을 누르면 아래 안내를 표시하고 편집 화면으로 이동해도 됩니다.

```txt
최종 제출이 취소되었습니다.
```

# 7. 제출 버튼

제출 확인/설문 화면 하단에 제출 버튼을 배치해 주세요.

버튼 문구:

```txt
제출
```

동작:

```txt
질문 1 답변 있음
질문 2에서 네 선택
→ 실제 최종 제출 진행
→ 제출 완료 화면으로 이동
```

필수값이 없으면 제출하지 않고 해당 질문 근처 또는 상단에 에러 문구를 표시해 주세요.

# 8. 실제 저장 데이터에 추가할 항목

최종 제출 시 기존에 저장하던 메뉴 구조/변경 로그는 그대로 유지해 주세요.

추가로 아래 값을 함께 저장해 주세요.

```js
{
  intention: "사용자가 입력한 메뉴 구성 의도",
  finalConfirm: "yes",
  submittedAt: "제출 시각"
}
```

필드명은 기존 서버/Supabase 구조에 맞게 조정해도 됩니다.

권장 필드명:

```txt
intention
final_confirm
submitted_at
```

또는 기존 camelCase 규칙을 사용 중이라면:

```txt
intention
finalConfirm
submittedAt
```

중요한 점은 기존 제출 데이터에 사용자의 마지막 의견이 함께 저장되어야 한다는 것입니다.

## 기대 동작

```txt

기존 미리보기 기능을 재사용해 주세요.

새로운 미리보기 로직을 만들기보다는, 현재 메뉴 상태를 기준으로 기존 미리보기 모달/화면을 열면 됩니다.

# 11. 화면 상태 관리 제안

현재 화면 상태를 아래처럼 관리할 수 있습니다.

```js
const [submitStep, setSubmitStep] = useState("edit");
```

상태 예시:

```txt
edit
submitForm
submitted
```

흐름:

```js
최종 제출 버튼 클릭
→ setSubmitStep("submitForm")

제출 확인 화면에서 제출 성공
→ setSubmitStep("submitted")

취소
→ setSubmitStep("edit")
```

또는 라우팅 구조를 사용 중이면 별도 route로 분리해도 됩니다.

예시:

```txt
/edit
/submit
/submitted
```

단, 현재 구조를 크게 바꾸지 않고 상태 기반으로 처리하는 것이 더 안전해 보입니다.

# 12. 기존 최종 제출 함수 재사용

실제 서버 저장 로직은 기존 최종 제출 함수를 최대한 재사용해 주세요.

기존 함수가 예를 들어 아래와 같다면:

```js
handleFinalSubmit()
```

수정 후에는 confirm 없이 제출 화면에서 검증 후 호출합니다.

예시:

```js
async function handleSubmitForm() {
  if (!intention.trim()) {
    setError("메뉴 구성 의도를 입력해주세요.");
    return;
  }

  if (!finalConfirm) {
    setError("최종 제출 여부를 선택해주세요.");
    return;
  }

  if (finalConfirm === "cancel") {
    setSubmitStep("edit");
    return;
  }

  await handleFinalSubmit({
    intention,
    finalConfirm: "yes",
  });

  setSubmitStep("submitted");
}
```

중요:

```txt
기존 서버 저장 로직은 유지
기본 confirm만 제거
제출 전 설문 화면을 거쳐 저장
```

# 13. 제출 중 로딩 처리

제출 버튼을 누른 뒤 서버 저장 중에는 중복 제출을 막아 주세요.

```txt
제출 중...
```

또는 버튼 disabled 처리:

```jsx
<button disabled={isSubmitting}>
  {isSubmitting ? "제출 중..." : "제출"}
</button>
```

제출 실패 시에는 완료 화면으로 넘어가지 말고 에러 메시지를 보여주세요.

예시:

```txt
제출 중 오류가 발생했습니다. 잠시 후 다시 시도해주세요.
```

# 14. 취소 처리

제출 확인 화면에서 사용자가 `취소`를 선택하고 제출 버튼을 누르면 실제 최종 제출을 하지 않습니다.

동작은 아래 중 하나로 처리해 주세요.

권장:

```txt
취소 선택 후 제출 클릭
→ 편집 화면으로 돌아감
```

이때 메뉴 편집 상태는 그대로 유지되어야 합니다.

# 핵심 요약

최종 제출 시 브라우저 기본 confirm 팝업을 사용하지 말고,
별도의 제출 확인/설문 화면을 보여주세요.

제출 전 사용자가 아래 두 가지를 입력/선택하게 해 주세요.

```txt
1. 메뉴 구성 의도
2. 위 메뉴 배치로 최종 제출하시겠습니까? [네/취소]
```

`네`를 선택하고 제출하면 기존 최종 제출 저장 로직을 실행하고,
성공 후에는 `참여해주셔서 감사합니다` 문구가 있는 제출 완료 화면으로 이동하게 해 주세요.

제출 완료 화면에서는 기존 미리보기 기능을 다시 열 수 있게 해 주세요.

```
