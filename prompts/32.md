# 리스트모드 Drag & Drop 개선 요청: @hello-pangea/dnd 적용

## 작업 목표

현재 메뉴 편집기의 **리스트모드 드래그앤드랍**을 더 안정적이고 자연스럽게 개선하고 싶습니다.

적용할 라이브러리는 다음입니다.

```txt
@hello-pangea/dnd
https://github.com/hello-pangea/dnd
````

이 라이브러리는 React 기반 리스트 드래그앤드랍에 적합하며,
리스트 아이템 이동 시 placeholder, 자연스러운 밀림 효과, 자동 스크롤, 드래그 핸들 지정 등을 안정적으로 처리할 수 있습니다.

---

이번 작업은 **리스트모드에만 적용**해 주세요.

```txt
적용 대상: 리스트 편집 모드
```

## 기대 개선점

리스트모드에서 다음 부분이 개선되어야 합니다.

```txt
1. 드래그 중 아이템 이동이 더 부드럽게 보일 것
2. 드롭 위치 placeholder가 정확하게 표시될 것
3. 기존 아이템이 자연스럽게 밀릴 것
4. 드래그 중 보이는 위치와 실제 드롭 결과가 일치할 것
5. 긴 리스트에서 자동 스크롤이 안정적으로 동작할 것
6. 드래그 핸들 영역이 명확할 것
7. 같은 레벨 내 순서 변경이 안정적일 것
8. 다른 부모로 이동할 때 위치 계산이 정확할 것
```

# 메뉴 레벨 구조

현재 메뉴는 3단계 구조입니다.

```txt
Level 1
 └ Level 2
    └ Level 3
```

각 레벨의 규칙은 반드시 유지해야 합니다.

```txt
Level 1은 최상위에만 존재 가능
Level 2는 Level 1 아래에만 존재 가능
Level 3은 Level 2 아래에만 존재 가능
```

---

# 허용 동작

리스트모드에서 허용되는 동작은 다음입니다.

```txt
Level 1끼리 순서 변경
Level 2를 같은 Level 1 내부에서 순서 변경
Level 2를 다른 Level 1 아래로 이동
Level 3을 같은 Level 2 내부에서 순서 변경
Level 3을 다른 Level 2 아래로 이동
```

---

# 금지 동작

아래 동작은 반드시 막아야 합니다.

```txt
Level 1을 Level 2 아래로 이동
Level 1을 Level 3 아래로 이동
Level 2를 Level 2의 자식으로 이동
Level 2를 Level 3 아래로 이동
Level 3을 Level 1 바로 아래로 이동
Level 3을 Level 3의 자식으로 이동
```

# 구현 방향

`@hello-pangea/dnd`의 다음 컴포넌트를 사용해 리스트모드만 감싸 주세요.

```jsx
import {
  DragDropContext,
  Droppable,
  Draggable,
} from "@hello-pangea/dnd";
```

기본 구조는 다음과 같습니다.

```jsx
<DragDropContext onDragEnd={handleListDragEnd}>
  {/* Level 1 Droppable */}
  <Droppable droppableId="level-1-root" type="LEVEL_1">
    {(provided) => (
      <div ref={provided.innerRef} {...provided.droppableProps}>
        {level1Items.map((item, index) => (
          <Draggable
            key={item.id}
            draggableId={item.id}
            index={index}
          >
            {(provided) => (
              <div
                ref={provided.innerRef}
                {...provided.draggableProps}
              >
                <div {...provided.dragHandleProps}>
                  드래그 핸들
                </div>

                메뉴 카드
              </div>
            )}
          </Draggable>
        ))}

        {provided.placeholder}
      </div>
    )}
  </Droppable>
</DragDropContext>
```

실제 프로젝트 구조에 맞게 컴포넌트는 분리해도 됩니다.

# Level별 Droppable 분리

레벨별로 Droppable `type`을 분리해 주세요.

```txt
LEVEL_1
LEVEL_2
LEVEL_3
```

예시:

```jsx
<Droppable droppableId="level-1-root" type="LEVEL_1">
```

```jsx
<Droppable droppableId={`level-2-parent-${level1Id}`} type="LEVEL_2">
```

```jsx
<Droppable droppableId={`level-3-parent-${level2Id}`} type="LEVEL_3">
```

이렇게 하면 Level 2가 Level 3 영역으로 들어가거나,
Level 3이 Level 1 영역으로 들어가는 문제를 구조적으로 줄일 수 있습니다.

---

# droppableId 규칙

`droppableId`는 나중에 부모를 추출하기 쉬운 형태로 만들어 주세요.

권장 형식:

```txt
level-1-root
level-2-parent-{level1Id}
level-3-parent-{level2Id}
```

예시:

```txt
level-2-parent-L01_STUDENT
level-3-parent-L02_STAR
```

드롭 시에는 `destination.droppableId`에서 부모 ID를 추출해
어느 부모 아래로 이동하는지 계산해야 합니다.

---

# Draggable ID 규칙

`draggableId`는 메뉴 아이템의 고유 ID를 사용해 주세요.

```jsx
<Draggable draggableId={item.id} index={index}>
```

만약 현재 메뉴 코드가 고유값이라면 `code`를 사용해도 됩니다.

```jsx
<Draggable draggableId={item.code} index={index}>
```

단, 전체 메뉴에서 중복되지 않는 값이어야 합니다.

# onDragEnd 처리 기준

`onDragEnd`에서는 반드시 아래 값을 확인해야 합니다.

```txt
source.droppableId
destination.droppableId
source.index
destination.index
draggableId
type
dragged item level
source parent
destination parent
```

기본 형태:

```js
function handleListDragEnd(result) {
  const { source, destination, draggableId, type } = result;

  if (!destination) return;

  if (
    source.droppableId === destination.droppableId &&
    source.index === destination.index
  ) {
    return;
  }

  const draggedItem = findMenuItemById(draggableId);

  if (!draggedItem) return;

  const nextMenuState = calculateNextMenuStateForListDnd({
    draggedItem,
    type,
    source,
    destination,
    currentMenuState: menuState,
  });

  if (!nextMenuState) {
    return;
  }

  commitMenuChange(nextMenuState);
}
```

---

# 실제 상태 변경은 commitMenuChange만 사용

드래그앤드랍 성공 시 실제 메뉴 상태 변경은 반드시 기존 공통 함수로만 처리해 주세요.

```js
commitMenuChange(nextMenuState);
```

직접 `setMenuState()`를 호출하거나,
리스트모드 전용 히스토리 함수를 새로 만들지 말아 주세요.

나쁜 예:

```js
setMenuState(nextMenuState);
saveListHistory(nextMenuState);
```

좋은 예:

```js
commitMenuChange(nextMenuState);
```

# 히스토리 연동 규칙

히스토리는 기존 undo/redo 구조를 그대로 사용해야 합니다.

드롭 성공 시에만 히스토리에 1회 저장되어야 합니다.

```txt
드래그 시작 → 히스토리 변화 없음
드래그 중 → 히스토리 변화 없음
드롭 취소 → 히스토리 변화 없음
같은 위치에 드롭 → 히스토리 변화 없음
잘못된 위치 → 히스토리 변화 없음
드롭 성공 → commitMenuChange(nextMenuState) 1회 호출
```

# 같은 위치 드롭 처리

아래 경우에는 아무 작업도 하지 않아야 합니다.

```js
if (
  source.droppableId === destination.droppableId &&
  source.index === destination.index
) {
  return;
}
```

이때 히스토리도 증가하면 안 됩니다.

# Level 1 처리

Level 1은 최상위 목록 안에서만 이동 가능합니다.

```txt
Droppable: level-1-root
type: LEVEL_1
```

Level 1 이동은 Level 1 리스트 안에서의 순서 변경만 허용합니다.

```txt
허용:
Level 1 순서 변경

금지:
Level 1을 Level 2 아래로 이동
Level 1을 Level 3 아래로 이동
```

`@hello-pangea/dnd`의 `type`을 분리하면
Level 1이 다른 레벨 Droppable에 들어가지 않도록 할 수 있습니다.

---

# Level 2 처리

Level 2는 Level 1 아래에만 존재해야 합니다.

각 Level 1마다 Level 2 Droppable을 만들어 주세요.

```jsx
<Droppable droppableId={`level-2-parent-${level1.id}`} type="LEVEL_2">
```

허용 동작:

```txt
같은 Level 1 내부에서 Level 2 순서 변경
다른 Level 1 아래로 Level 2 이동
```

예시:

```txt
학사정보 > 학사관리
```

를

```txt
star 학생지원
```

아래로 이동할 수 있어야 합니다.

이 경우 `parentId`가 기존 Level 1에서 새 Level 1로 바뀌고,
destination.index 위치에 삽입되어야 합니다.

---

# Level 3 처리

Level 3은 Level 2 아래에만 존재해야 합니다.

각 Level 2마다 Level 3 Droppable을 만들어 주세요.

```jsx
<Droppable droppableId={`level-3-parent-${level2.id}`} type="LEVEL_3">
```

허용 동작:

```txt
같은 Level 2 내부에서 Level 3 순서 변경
다른 Level 2 아래로 Level 3 이동
```

예시:

```txt
star 관리 > 희망학과 선택
```

을

```txt
장학관리
```

아래로 이동할 수 있어야 합니다.

이 경우 `parentId`가 기존 Level 2에서 새 Level 2로 바뀌고,
destination.index 위치에 삽입되어야 합니다.

---

# Level 타입 불일치 차단

`type`이 다르면 기본적으로 다른 Droppable에 드롭되지 않게 구성해 주세요.

의도:

```txt
LEVEL_1 Draggable → LEVEL_1 Droppable만 허용
LEVEL_2 Draggable → LEVEL_2 Droppable만 허용
LEVEL_3 Draggable → LEVEL_3 Droppable만 허용
```

이렇게 하면 잘못된 레벨 이동을 복잡한 금지 표시 없이 구조적으로 막을 수 있습니다.

---

# 접힘/펼침 상태와 Droppable

접힌 메뉴의 하위 목록은 화면에 보이지 않을 수 있습니다.

이 경우 원칙은 다음과 같습니다.

```txt
보이는 하위 목록만 Droppable로 사용
접힌 하위 목록은 드롭 대상에서 제외
```

다만 Level 2 또는 Level 3을 다른 부모로 이동하려면
해당 부모가 펼쳐진 상태에서만 정확한 위치 선택이 가능하도록 해 주세요.

자동 펼침이 필요하다면 추후 별도 작업으로 처리하고,
이번 작업에서는 무리하게 자동 펼침까지 구현하지 않아도 됩니다.

# 보기 옵션과 히스토리

다음 동작은 메뉴 구조 변경이 아니므로 히스토리에 저장하면 안 됩니다.

```txt
Level 1만 보기
Level 2까지 보기
전체 펼치기
메뉴 접기
메뉴 펼치기
드래그 hover
선택 메뉴 변경
편집 탭 전환
```

히스토리에 저장해야 하는 것은 실제 메뉴 구조 변경뿐입니다.

```txt
메뉴 순서 변경
메뉴 부모 변경
메뉴 추가
메뉴 삭제
메뉴 이름 변경
메뉴 레벨 변경
```

# placeholder 표시

`@hello-pangea/dnd`에서 제공하는 placeholder를 반드시 렌더링해 주세요.

```jsx
{provided.placeholder}
```

드롭될 위치를 사용자가 정확히 알 수 있습니다.

# 드래그 핸들

카드 전체가 드래그되지 않아도 됩니다.

기존에 있는 점 6개 아이콘 또는 드래그 핸들 영역에만
`dragHandleProps`를 적용해 주세요.

예시:

```jsx
<div {...provided.dragHandleProps} className="drag-handle">
  ⋮⋮
</div>
```

이렇게 하면 버튼 클릭, 이름 수정, 펼침/접힘 클릭과 드래그가 충돌하지 않습니다.

---

# 버튼 클릭과 드래그 충돌 방지

메뉴 카드 안에는 다음 버튼들이 있습니다.

```txt
수정
추가
삭제
펼침/접힘
```

이 버튼들은 드래그와 충돌하면 안 됩니다.

따라서 `dragHandleProps`는 카드 전체가 아니라
드래그 핸들에만 적용하는 것을 권장합니다.

# 자동 스크롤

리스트가 길어질 경우 드래그 중 자동 스크롤이 자연스럽게 동작해야 합니다.

`@hello-pangea/dnd`의 기본 자동 스크롤 기능을 최대한 활용해 주세요.

만약 현재 리스트 컨테이너가 내부 스크롤 구조라면
Droppable이 해당 스크롤 영역 안에서 정상 동작하도록 ref와 overflow 구조를 확인해 주세요.

# SSR / Next.js 주의

Next.js 환경에서 hydration 문제가 생기지 않도록 주의해 주세요.

`@hello-pangea/dnd`는 클라이언트에서 동작해야 하므로,
리스트 편집 컴포넌트가 Client Component인지 확인해 주세요.

파일 상단에 필요하면 다음을 유지해 주세요.

```js
"use client";
```

또한 서버 렌더링 시점과 클라이언트 렌더링 시점의 메뉴 구조가 달라지지 않도록,
localStorage나 window 기반 상태를 렌더링 중 직접 사용하지 말아 주세요.

# 데이터 변경 계산 함수

드롭 결과를 실제 메뉴 데이터에 반영하는 함수는 별도로 분리해도 됩니다.

예시:

```js
function calculateNextMenuStateForListDnd({
  draggedItem,
  type,
  source,
  destination,
  currentMenuState,
}) {
  // 1. source parent 확인
  // 2. destination parent 확인
  // 3. draggedItem level 확인
  // 4. 허용 가능한 이동인지 검사
  // 5. 기존 위치에서 제거
  // 6. destination.index에 삽입
  // 7. parentId 갱신
  // 8. order 재정렬
  // 9. nextMenuState 반환
}
```

유효하지 않은 이동이면 `null`을 반환해 주세요.

```js
if (!isValidMove) {
  return null;
}
```

# 같은 부모 안 이동 시 index 보정

같은 부모 안에서 아이템을 이동할 때는
기존 아이템을 제거한 뒤 삽입해야 하므로 index 보정에 주의해 주세요.

예시:

```txt
기존 순서: A B C D
B를 D 뒤로 이동
```

처리:

```txt
1. B 제거 → A C D
2. D 뒤에 B 삽입 → A C D B
```

같은 부모 안에서 source.index가 destination.index보다 작을 때
삽입 위치가 한 칸 밀릴 수 있으므로 실제 구현 구조에 맞게 보정해 주세요.

중요한 점은 다음입니다.

```txt
드래그 중 placeholder가 보여준 위치와 실제 드롭 결과가 일치해야 합니다.
```
