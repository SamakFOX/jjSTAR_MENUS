# 메뉴 배치 테스트 웹 프로젝트 기획서

## 1. 프로젝트 개요

본 프로젝트는 실제 사용자/테스터가 웹 화면에서 메뉴 구조를 직접 배치해보도록 하고, 그 결과를 수집하여 향후 데모 웹 또는 실제 서비스의 메뉴 구성에 반영하기 위한 테스트용 웹이다.

사용자는 제공된 메뉴 항목을 클릭, 드래그, 이동 버튼 등을 통해 원하는 위치와 계층 구조로 재배치한 후 최종 제출한다. 제출된 데이터는 관리자 또는 운영자가 확인하여 메뉴 위치, 메뉴 레벨, 사용자 선호 흐름 등을 분석하는 데 활용한다.

---

## 2. 선택한 기술 방향

본 프로젝트는 `React + Vercel` 기반으로 진행한다.

### 선택 이유

- 드래그 앤 드롭 UI 구현에 유리함
- 메뉴 계층 구조와 상태 관리가 쉬움
- 편집모드, 흔들림 모션, 선택 상태, 미리보기 등을 구현하기 좋음
- GitHub 연동 배포가 편리함
- 프론트 중심의 UX 테스트 도구 제작에 적합함
- 추후 Supabase, Firebase 등 외부 DB와 쉽게 연동 가능함

* 프로젝트 목적 안내

* 삭제 또는 되돌리기 버튼 표시

* 최종 배치 결과 요약
* 필수 항목 누락 여부 확인

* 제출 완료 후 안내

### 5) 관리자 화면

* 제출자 목록
* 제출 시간
* 메뉴 배치 결과
* 보상 지급 여부
* 데이터 다운로드

### 드래그 앤 드롭

```txt
@dnd-kit/core
@dnd-kit/sortable
```

또는

```txt
react-beautiful-dnd
```

단, 신규 프로젝트에서는 `@dnd-kit` 사용을 우선 고려한다.

초기에는 React 기본 상태 관리로 시작한다.

```txt

useReducer
useContext
```

메뉴 구조가 복잡해지면 다음을 고려한다.

```txt
zustand
```

### 데이터 저장

초기 후보:

```txt
Supabase
Firebase Firestore
```

간단한 제출 저장용이면 Supabase를 우선 고려한다.

## 9. 제한사항

### Vercel 관련 제한사항

* 서버를 장시간 실행하는 방식에는 적합하지 않음
* 파일을 서버 내부에 직접 저장하는 구조는 적합하지 않음
* 제출 데이터는 외부 DB에 저장해야 함
* 백엔드 기능은 API Route 또는 외부 서비스와 연동해야 함
* 무료 요금제 사용 시 트래픽, 빌드, 서버리스 함수 사용량 제한을 고려해야 함

### React/Next.js 관련 제한사항

* 드래그 앤 드롭 구현 시 상태 구조 설계가 중요함
* 메뉴 계층이 깊어질수록 데이터 변경 로직이 복잡해질 수 있음
* 모바일 터치 드래그 대응을 반드시 테스트해야 함
* 브라우저별 드래그 동작 차이를 확인해야 함
* 제출 전 데이터 검증 로직이 필요함

### 데이터 저장 관련 제한사항

* Vercel 자체에 DB가 있는 것이 아니므로 외부 DB 필요
* Supabase 또는 Firebase 계정 필요
* 제출 데이터가 많아지면 요금제 확인 필요
* 개인정보를 받을 경우 개인정보 처리 안내 필요
* 보상 지급을 위한 연락처 수집 시 보안 관리 필요

### UX 관련 제한사항

* 드래그만 제공하면 사용자가 어려워할 수 있음
* 반드시 버튼식 이동 기능도 함께 제공하는 것이 좋음
* 메뉴 레벨 변경 방식은 직관적이어야 함
* 제출 전 결과 확인 화면이 필요함
* 모바일 사용자를 고려한 터치 영역 확보 필요
* 너무 많은 메뉴를 한 화면에 보여주면 사용성이 떨어질 수 있음

## 10. 구현 시 주의사항

### 메뉴 ID는 절대 변경하지 않기

메뉴 이름은 바뀔 수 있지만, 분석을 위해 `id`는 고정해야 한다.

```js
{
  id: "L3-COURSE-APPLY",
  title: "수강신청"
}
```

### 제출 데이터는 원본 메뉴와 분리하기

사용자가 수정한 결과는 원본 메뉴 데이터와 별도로 저장한다.

```txt
initialMenu: 원본 메뉴
menuResult: 사용자가 제출한 결과
```

### 되돌리기 기능 고려

사용자가 실수했을 때 복구할 수 있어야 한다.

추천 기능:

```txt
초기화
이전 단계로 되돌리기
선택 항목 원위치
```

### 드래그 외 대체 조작 제공

드래그가 어려운 사용자를 위해 버튼식 조작도 제공한다.

예시:

```txt
위로 이동
아래로 이동
상위 메뉴로 이동
하위 메뉴로 이동
왼쪽으로 이동
오른쪽으로 이동
```

## 13. 최종 판단

이 프로젝트는 단순한 정적 페이지가 아니라 사용자가 직접 메뉴를 조작하고 제출하는 UX 테스트 도구이다.

따라서 `HTML + iwinv` 방식도 가능하지만, 드래그 앤 드롭, 메뉴 계층 관리, 편집모드, 미리보기, 제출 검증 등을 고려하면 `React + Vercel` 방식이 더 적합하다.

특히 초기 개발 속도, UI 조작감, 유지보수성을 고려하면 다음 구조를 우선 추천한다.

```txt
React / Next.js
+ Vercel
+ Supabase
+ @dnd-kit
```

```

요약하면 이 프로젝트는 **“설문폼”이 아니라 “메뉴 구조를 직접 편집하는 테스트 도구”**라서 React로 가는 게 맞아.
