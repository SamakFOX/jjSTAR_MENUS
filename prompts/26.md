# 메뉴 배치 테스트 UI/기능 수정 요청

## 1. 입장 화면에서 이름/닉네임 입력 제거

현재 입장 화면에는 다음 입력 항목이 있습니다.

- 인증코드
- 이름 또는 닉네임

하지만 이 테스트의 목적은 “JJSTAR 메뉴 배치 테스트 참여”이므로, 이름 또는 닉네임 입력은 필수로 보이지 않습니다.

사용자가 불필요하게 개인정보 입력으로 느낄 수 있고, 실제 기능상 참여자 식별이 꼭 필요하지 않다면 제거하는 방향이 좋겠습니다.

### 수정 요청

입장 화면에서 `이름 또는 닉네임` 입력란을 제거해 주세요.

남길 항목은 다음 하나만 있으면 됩니다.

- 인증코드

### 안내 문구도 수정

기존 문구 예시:

```txt
인증은 서버에서 처리되며, 인증 정보는 현재 브라우저 탭에서만 유지됩니다.
````

수정 문구 예시:

```txt
발급받은 인증코드를 입력하면 메뉴 배치 테스트에 참여할 수 있습니다.
인증 정보는 현재 브라우저에서만 유지됩니다.
```

또는 더 간단하게:

```txt
인증코드는 테스트 참여 확인 용도로만 사용됩니다.
```

## 2. 리스트 편집 / 화면에서 편집 간 히스토리 공유 문제

현재 리스트 편집 모드에서는 되돌리기 기능이 정상적으로 동작합니다.

하지만 화면에서 편집, 즉 비주얼 모드에서 메뉴를 이동하거나 수정한 뒤 `X` 버튼으로 나가면 변경 사항은 실제로 적용되지만, 되돌리기 카운트에는 반영되지 않는 문제가 있습니다.

### 현재 문제 예시

예를 들어 다음과 같은 상황입니다.

1. 리스트 편집에서 5번 수정
2. 화면에서 편집, 즉 비주얼 모드에서 5번 수정
3. 다시 리스트 편집으로 돌아옴
4. 되돌리기 버튼에는 여전히 5회만 표시됨

하지만 실제로는 총 10번의 수정 작업이 있었으므로, 되돌리기 가능 횟수는 10회가 되어야 합니다.

현재 구조에서는 비주얼 모드에서 작업한 변경 내역이 히스토리 카운트에 반영되지 않아서, 되돌리기를 5번 눌러도 비주얼 모드 작업만 되돌아가고, 리스트 모드에서 작업했던 이전 5개의 변경까지 되돌아가지 못합니다.

## 3. 히스토리 관리 방식 통합 요청

리스트 편집 모드와 화면에서 편집 모드가 서로 다른 히스토리를 가지면 안 됩니다.

두 모드는 같은 메뉴 데이터를 편집하는 것이므로, 하나의 공통 히스토리 스택을 사용해야 합니다.

### 수정 요청

다음 작업들은 모두 동일한 히스토리 스택에 저장되도록 해 주세요.

* 리스트 편집에서 메뉴 이동
* 리스트 편집에서 메뉴 이름 변경
* 리스트 편집에서 메뉴 추가
* 리스트 편집에서 메뉴 삭제
* 화면에서 편집 모드에서 메뉴 이동
* 화면에서 편집 모드에서 메뉴 순서 변경
* 화면에서 편집 모드에서 메뉴 표시 구조 변경

즉, 사용자가 리스트 편집과 화면에서 편집을 왔다 갔다 하더라도 되돌리기/다시실행이 전체 작업 흐름 기준으로 정확하게 동작해야 합니다.

## 4. 되돌리기 카운트 정상 반영

현재 되돌리기 버튼은 다음처럼 표시됩니다.

```txt
되돌리기 (5)
```

비주얼 모드에서 작업한 내용도 이 카운트에 포함되어야 합니다.

### 기대 동작

예시:

| 작업                | 기대되는 되돌리기 카운트 |
| ----------------- | ------------: |
| 리스트 편집에서 1회 수정    |      되돌리기 (1) |
| 리스트 편집에서 5회 수정    |      되돌리기 (5) |
| 이후 비주얼 모드에서 1회 수정 |      되돌리기 (6) |
| 이후 비주얼 모드에서 5회 수정 |     되돌리기 (10) |

리스트 편집과 비주얼 편집을 오가더라도 카운트가 누락되지 않아야 합니다.

## 5. 다시실행 기능 추가 요청

되돌리기 기능만 있으면 작업을 되돌린 뒤 다시 복구할 방법이 부족합니다.

따라서 `다시실행` 기능을 추가해 주세요.

### 버튼 구성

현재:

```txt
[되돌리기 (5)]
```

수정 후:

```txt
[되돌리기 (5)] [다시실행 (0)]
```

또는 더 직관적으로:

```txt
[↶ 되돌리기 (5)] [↷ 다시실행 (0)]
```

### 기대 동작

1. 사용자가 메뉴를 수정하면 되돌리기 카운트 증가
2. 되돌리기를 누르면 수정 전 상태로 돌아감
3. 이때 다시실행 카운트 증가
4. 다시실행을 누르면 방금 되돌린 작업이 다시 적용됨
5. 되돌린 상태에서 새로운 편집을 하면 기존 다시실행 히스토리는 초기화

## 6. 히스토리 최대 저장 개수

히스토리는 최대 20회까지 저장하면 좋겠습니다.

리소스를 많이 먹지 않는 구조라면 20회로 적용해 주세요.

### 요청 사항

* 최대 히스토리 저장 개수: 20회
* 20회를 초과하면 가장 오래된 히스토리부터 제거
* 리스트 편집과 비주얼 편집의 작업을 합산하여 20회 기준으로 관리

### 안내 문구 추가

되돌리기/다시실행 버튼 바로 위 또는 근처에 다음 안내 문구를 추가해 주세요.

```txt
최근 작업 내역은 최대 20회까지 저장됩니다.
```

또는 더 자연스럽게:

```txt
최근 20회의 편집 내역을 되돌리거나 다시 실행할 수 있습니다.
```

## 7. 되돌리기 후 토스트의 “실행 취소” 버튼 제거

현재 되돌리기를 누르면 하단 토스트에 다음과 같은 문구가 표시됩니다.

```txt
작업이 취소되었습니다. 실행 취소
```

하지만 이 상황에서 `실행 취소`라는 버튼은 사용자가 헷갈릴 수 있습니다.

사용자 입장에서는 다음처럼 인식될 수 있습니다.

* 방금 누른 되돌리기를 취소하는 것인지
* 원래 작업을 취소하는 것인지
* 다시실행과 같은 기능인지
* 상단의 되돌리기 버튼과 어떤 차이가 있는지

이미 상단에 `되돌리기` 기능이 존재하므로, 토스트 안의 `실행 취소` 버튼은 제거하는 것이 더 직관적입니다.

### 수정 요청

되돌리기 후 토스트는 버튼 없이 문구만 표시해 주세요.

예시:

```txt
작업이 취소되었습니다.
```

다시실행 후에는 다음처럼 표시하면 됩니다.

```txt
작업이 다시 실행되었습니다.
```

## 8. 상단 편집 도구 영역 고정 요청

현재 리스트 모드에서 상단의 두 영역이 스크롤에 따라 화면 밖으로 사라질 수 있습니다.

수정할 메뉴가 많을 때는 사용자가 되돌리기, 다시실행, 편집 완료, 초기화, 모드 전환 등을 자주 사용해야 하므로 상단 도구 영역이 항상 보이는 것이 좋습니다.

### 고정 대상

리스트 모드에서 다음 두 섹션을 고정해 주세요.

1. 상단 편집 상태 영역

```txt
[편집 완료] [리스트 편집] [화면에서 편집] [초기화] [미리보기] [최종 제출]
```

2. 되돌리기/도구 영역

```txt
[Level 1 추가] [Level 1 순서변경] [되돌리기] [다시실행] [Level 1만 보기] [Level 2까지 보기] [전체 펼치기]
```

### 기대 동작

* 리스트가 길어져 아래로 스크롤해도 위 두 섹션은 계속 화면 상단에 고정
* 사용자가 언제든 되돌리기/다시실행/편집 완료를 사용할 수 있음
* 메뉴 항목 리스트만 아래에서 스크롤됨
* 고정 영역이 콘텐츠를 가리지 않도록 적절한 배경, 그림자, z-index 적용

### 스타일 참고

고정 영역은 다음처럼 처리하면 좋겠습니다.

```css
position: sticky;
top: 0;
z-index: 50;
background: #f5f6fb;
box-shadow: 0 2px 8px rgba(15, 23, 42, 0.08);
```

두 번째 도구 영역은 첫 번째 영역 아래에 붙어야 하므로 `top` 값을 첫 번째 영역 높이에 맞춰 조정해 주세요.

## 9. 비주얼 모드 종료 시 히스토리 누락 방지

비주얼 모드에서 `X` 버튼으로 나갈 때 변경사항이 적용되는 구조라면, 그 시점에 히스토리가 한 번에 저장되는 것이 아니라 실제 편집 단위별로 저장되어야 합니다.

### 나쁜 동작

```txt
비주얼 모드에서 여러 번 이동
→ X로 나감
→ 최종 결과만 반영
→ 히스토리 카운트는 증가하지 않거나 1회만 증가
```

### 기대 동작

```txt
비주얼 모드에서 메뉴 A 이동
→ 히스토리 +1

비주얼 모드에서 메뉴 B 이동
→ 히스토리 +1

비주얼 모드에서 메뉴 C 이동
→ 히스토리 +1

X로 나감
→ 이미 저장된 히스토리 유지
```

즉, 비주얼 모드 작업도 리스트 편집과 동일하게 “작업 단위”로 히스토리에 저장되어야 합니다.

## 10. 구현 방향 제안

히스토리 구조는 다음처럼 공통으로 관리하는 방식이 좋습니다.

```js
const HISTORY_LIMIT = 20;

const [menuState, setMenuState] = useState(initialMenuState);
const [undoStack, setUndoStack] = useState([]);
const [redoStack, setRedoStack] = useState([]);
```

메뉴 변경 전 현재 상태를 undoStack에 저장합니다.

```js
const commitMenuChange = (nextState) => {
  setUndoStack((prev) => {
    const nextUndoStack = [...prev, menuState];
    return nextUndoStack.slice(-HISTORY_LIMIT);
  });

  setRedoStack([]);
  setMenuState(nextState);
};
```

되돌리기:

```js
const handleUndo = () => {
  setUndoStack((prevUndo) => {
    if (prevUndo.length === 0) return prevUndo;

    const previousState = prevUndo[prevUndo.length - 1];

    setRedoStack((prevRedo) => {
      const nextRedoStack = [...prevRedo, menuState];
      return nextRedoStack.slice(-HISTORY_LIMIT);
    });

    setMenuState(previousState);

    return prevUndo.slice(0, -1);
  });
};
```

다시실행:

```js
const handleRedo = () => {
  setRedoStack((prevRedo) => {
    if (prevRedo.length === 0) return prevRedo;

    const nextState = prevRedo[prevRedo.length - 1];

    setUndoStack((prevUndo) => {
      const nextUndoStack = [...prevUndo, menuState];
      return nextUndoStack.slice(-HISTORY_LIMIT);
    });

    setMenuState(nextState);

    return prevRedo.slice(0, -1);
  });
};
```

주의할 점:

* 리스트 모드와 비주얼 모드가 각각 별도 state를 갖지 않도록 해 주세요.
* 실제 메뉴 데이터는 하나의 source of truth를 사용해야 합니다.
* 리스트 모드와 비주얼 모드는 같은 `menuState`를 다른 UI로 보여주는 구조가 되어야 합니다.
* 비주얼 모드에서 수정한 내용도 반드시 `commitMenuChange()` 같은 공통 변경 함수를 통해 반영해야 합니다.

## 핵심 요약

현재 문제는 리스트 편집과 비주얼 편집의 변경 이력이 동일한 히스토리로 관리되지 않아 발생하는 것으로 보입니다.

다음 방향으로 수정해 주세요.

1. 입장 화면에서 이름/닉네임 입력 제거
2. 리스트 편집과 비주얼 편집의 히스토리 통합
3. 비주얼 모드 작업도 되돌리기 카운트에 반영
4. 다시실행 기능 추가
5. 전체 히스토리는 최대 20회 저장
6. 되돌리기/다시실행 버튼 근처에 히스토리 안내 문구 추가
7. 되돌리기 토스트의 “실행 취소” 버튼 제거
8. 리스트 모드 상단의 편집 도구 영역 2개를 sticky로 고정
9. 메뉴 데이터는 하나의 공통 state/source of truth로 관리

```
