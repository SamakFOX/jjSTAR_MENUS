# 비주얼모드 드래그 제한사항 적용 및 UI 탭 방식 변경 요청

## 1. 현재 문제

리스트모드는 현재 거의 정상적으로 동작하고 있습니다.

하지만 비주얼모드에서는 아직 드래그 제한사항이 제대로 적용되지 않아 다음 문제가 발생합니다.

```txt
1. Level 2 메뉴를 드래그해서 위치를 바꾸거나 다른 Level 1로 이동하려고 함
2. 그런데 의도와 다르게 다른 Level 2 안으로 들어감
3. Level 2가 Level 2의 자식처럼 들어가면서 구조가 깨짐
4. 결과적으로 Level 4까지 생기는 문제가 발생함
````

비주얼모드도 리스트모드와 동일한 구조 제한사항을 적용해야 합니다.

---

## 2. 비주얼모드에도 동일하게 적용할 핵심 구조 규칙

비주얼모드에서도 메뉴 구조는 반드시 다음 3단계까지만 허용합니다.

```txt
Level 1
  Level 2
    Level 3
```

허용 구조:

```txt
Level 1 = root에만 존재
Level 2 = Level 1의 children에만 존재
Level 3 = Level 2의 children에만 존재
```

금지 구조:

```txt
Level 1 → Level 2 하위로 이동 금지
Level 1 → Level 3 하위로 이동 금지

Level 2 → root로 이동 금지
Level 2 → Level 2 하위로 직접 이동 금지
Level 2 → Level 3 하위로 이동 금지

Level 3 → root로 이동 금지
Level 3 → Level 1 직접 하위로 이동 금지
Level 3 → Level 3 하위로 이동 금지

Level 4 이상 생성 금지
```

---

## 3. 비주얼모드 드래그 허용 규칙

비주얼모드에서도 리스트모드와 동일하게 다음 이동만 허용합니다.

| 드래그 대상  | 드롭 대상         | 처리                                           |
| ------- | ------------- | -------------------------------------------- |
| Level 1 | Level 1 정렬 영역 | Level 1 순서 변경만 허용                            |
| Level 2 | 같은 Level 1 내부 | Level 2 순서 변경                                |
| Level 2 | 다른 Level 1 영역 | 해당 Level 1의 children으로 이동                    |
| Level 2 | Level 2 영역    | 직접 자식 이동 금지, 통합 팝업 처리                        |
| Level 2 | Level 3 영역    | 금지                                           |
| Level 3 | 같은 Level 2 내부 | Level 3 순서 변경                                |
| Level 3 | 다른 Level 2 영역 | 해당 Level 2의 children으로 이동                    |
| Level 3 | Level 1 영역    | 금지                                           |
| Level 3 | Level 3 영역    | 같은 부모 안이면 순서 변경, 다른 부모면 금지 또는 Level 2 드롭존 사용 |

---

## 4. Level 2가 Level 2 안으로 들어가는 문제 수정

현재 비주얼모드에서는 Level 2를 드래그했을 때 다른 Level 2의 내부로 들어가면서 Level 4가 생기고 있습니다.

이 동작은 금지해야 합니다.

### 잘못된 결과

```txt
학사정보 L1
  수강관리 L2
    장학관리 L2
      장학 공지사항 L3
```

또는:

```txt
학사정보 L1
  수강관리 L2
    장학관리 L3
      장학 공지사항 L4
```

이런 구조는 절대 생성되면 안 됩니다.

---

## 5. Level 2 → Level 2 드롭 시 통합 팝업

리스트모드와 동일하게, Level 2를 다른 Level 2에 넣으려는 경우 직접 이동하지 말고 통합 팝업을 표시합니다.

문구:

```txt
"{이동할 Level 2 이름}"은 Level 2 메뉴이므로 "{대상 Level 2 이름}"의 하위 메뉴가 될 수 없습니다.

대신 "{이동할 Level 2 이름}" 메뉴를 제거하고,
그 하위메뉴를 "{대상 Level 2 이름}"의 하위메뉴로 통합할 수 있습니다.

진행하시겠습니까?
```

버튼:

```txt
[통합하기] [취소]
```

처리:

```txt
통합하기:
- 이동할 Level 2 자체는 제거
- 이동할 Level 2의 하위 Level 3만 대상 Level 2의 children으로 이동
- 화면에 [삭제됨] 그룹은 남기지 않음
- 변경 로그에 통합 기록 남김

취소:
- 아무 작업도 하지 않음
```

---

## 6. Level 2 → 다른 Level 1 이동

비주얼모드에서 Level 2를 다른 Level 1 영역으로 드래그하면 해당 Level 1의 children으로 이동해야 합니다.

예시:

```txt
변경 전

star 학생지원 L1
  장학관리 L2
    장학 공지사항 L3

학사정보 L1
  수강관리 L2
```

`장학관리 L2`를 `학사정보 L1` 영역으로 이동:

```txt
변경 후

star 학생지원 L1

학사정보 L1
  수강관리 L2
  장학관리 L2
    장학 공지사항 L3
```

이때 Level 2의 하위 Level 3도 그룹으로 함께 이동해야 합니다.

확인 문구:

```txt
"{Level 2 이름}"의 하위메뉴도 함께 이동하시겠습니까?
```

버튼:

```txt
[함께 이동] [취소]
```

---

## 7. Level 3 → 다른 Level 2 이동

비주얼모드에서 Level 3을 다른 Level 2 영역으로 드래그하면 해당 Level 2의 children으로 이동해야 합니다.

예시:

```txt
변경 전

학사정보 L1
  수강관리 L2
    수강신청 L3

star 학생지원 L1
  장학관리 L2
```

`수강신청 L3`을 `장학관리 L2` 영역으로 이동:

```txt
변경 후

학사정보 L1
  수강관리 L2

star 학생지원 L1
  장학관리 L2
    수강신청 L3
```

---

## 8. 비주얼모드 드롭존 구분

비주얼모드에서도 리스트모드처럼 “순서 정렬”과 “하위 이동/통합”을 명확히 분리해야 합니다.

권장 방식:

```txt
Level 1 카드 내부의 Level 2 영역
→ Level 2를 받을 수 있는 영역

Level 2 카드 내부의 Level 3 영역
→ Level 3을 받을 수 있는 영역

Level 2 카드 자체
→ Level 2 통합 팝업 대상

같은 부모 내부의 카드 사이
→ 순서 정렬
```

드래그 중에는 결과 문구를 표시합니다.

예시:

```txt
"학사정보" 하위로 이동
"수강관리"에 하위메뉴 통합
"장학관리" 하위로 이동
Level 3은 Level 1의 직접 하위로 이동할 수 없습니다.
```

---

## 9. 비주얼모드에서도 tree 기준 검증 필수

비주얼모드에서 이동 후 level 값을 강제로 보정하면 안 됩니다.

금지 예시:

```js
level: Math.max(1, Math.min(MAX_LEVEL, nextLevel))
```

이 방식은 잘못된 구조가 생긴 뒤 숫자만 맞추는 방식이라, Level 4나 잘못된 부모 구조가 발생할 수 있습니다.

반드시 이동 전에 tree 기준으로 검증해야 합니다.

```txt
Level 1 이동 가능 위치: root
Level 2 이동 가능 부모: Level 1
Level 3 이동 가능 부모: Level 2
```

조건에 맞지 않으면 이동을 취소하거나 통합 팝업을 표시합니다.

---

## 10. 리스트모드와 비주얼모드 동일 상태 사용

비주얼모드와 리스트모드는 반드시 같은 `menuTree`를 사용해야 합니다.

```txt
공통 원본 상태: menuTree

리스트모드:
- menuTree를 flatten해서 표시
- 리스트모드 제한사항 적용

비주얼모드:
- 같은 menuTree를 카드/박스 형태로 표시
- 리스트모드와 동일한 제한사항 적용
```

금지:

```txt
리스트모드용 state와 비주얼모드용 state를 따로 관리하지 않기
비주얼모드에서 별도 flat list를 만들어 부모를 추론하지 않기
```

# 11. 비주얼모드 UI 방식 변경 요청

## 11.1 현재 문제

현재 비주얼모드가 팝업 또는 모달처럼 표시되는 느낌입니다.

이 방식은 메뉴 배치 작업 중 화면 전환이 어색하고, 리스트모드와 비주얼모드가 분리된 기능처럼 느껴집니다.

비주얼모드는 별도 팝업이 아니라 기본 편집 화면의 탭처럼 사용할 수 있게 변경하는 것이 좋습니다.

---

## 11.2 원하는 방식

상단에 탭 형태로 모드를 전환합니다.

예시:

```txt
[리스트모드] [비주얼모드]
```

또는:

```txt
[리스트 편집] [비주얼 편집]
```

비주얼모드를 누르면 팝업이 뜨는 것이 아니라, 같은 편집 영역 안에서 화면만 전환됩니다.

---

## 11.3 탭 구조 예시

```txt
상단 편집 툴바

[메뉴 편집하기 / 편집 완료]
[리스트모드] [비주얼모드]
[Level 1 추가]
[Level 1 순서변경]
[되돌리기]
[Level 1만 보기] [Level 2까지 보기] [전체 펼치기]
```

본문 영역:

```txt
리스트모드 선택 시:
- 기존 리스트 편집 UI 표시

비주얼모드 선택 시:
- 동일 menuTree를 기반으로 비주얼 카드 UI 표시
```

---

## 11.4 상태 유지

모드 전환 시 메뉴 상태가 초기화되면 안 됩니다.

```txt
리스트모드에서 이동한 결과
→ 비주얼모드에 즉시 반영

비주얼모드에서 이동한 결과
→ 리스트모드에 즉시 반영
```

보기 상태도 가능하면 유지합니다.

```txt
현재 선택된 Level 1
열림/닫힘 상태
변경 로그
Undo stack
편집 중 상태
```

---

## 11.5 비주얼모드가 탭이 되어야 하는 이유

```txt
1. 팝업보다 작업 흐름이 자연스러움
2. 리스트모드와 같은 편집 기능이라는 인식이 쉬움
3. 메뉴 배치 작업 중 모드 전환이 빠름
4. 같은 menuTree를 사용한다는 구조가 명확해짐
5. 모달 닫기/열기 과정에서 상태가 꼬일 가능성이 줄어듦
```

# 12. 최종 요구사항 요약

```txt
1. 비주얼모드에도 리스트모드와 동일한 구조 제한사항을 적용한다.
2. Level 2가 Level 2 안으로 들어가 Level 4가 생기는 문제를 반드시 막는다.
3. Level 2 → Level 1 이동은 허용한다.
4. Level 2 → Level 2 이동은 직접 이동이 아니라 통합 팝업으로 처리한다.
5. Level 3 → Level 2 이동은 허용한다.
6. Level 3 → Level 1 이동은 금지한다.
7. 모든 이동 후 tree 구조 검증을 수행한다.
8. 비주얼모드와 리스트모드는 같은 menuTree를 사용한다.
9. 비주얼모드는 팝업/모달이 아니라 기본 편집 화면의 탭으로 제공한다.
10. 탭 전환 시 메뉴 상태, Undo, 로그, 편집 상태가 유지되어야 한다.
```
