# 메뉴 편집 화면 오류 수정 요청: 한글 깨짐 + 히스토리 중복 증가 문제

## 현재 문제

최근 수정 이후 리스트 편집 화면에서 다음 문제가 발생했습니다.

---

## 1. 한글 텍스트가 깨지는 문제

우측 히스토리/도구 박스와 일부 버튼, 토스트 문구에서 한글이 깨져서 표시됩니다.

예시로 다음과 같이 보입니다.

```txt
?껬쯔 ?꾜뎃
Level 1 ?뭣쨩?
?뭣됭ㅂㅁㄹ (1)
?쯔뭣?ㅆ?
???遺쯧쨩 硫쨩됨
````

이 문제는 실제 UI 문구가 깨진 상태로 렌더링되는 것으로 보입니다.

### 수정 요청

한글 문구가 정상적으로 표시되도록 수정해 주세요.

깨진 문자열을 모두 정상 한글 문구로 복구해 주세요.

---

## 2. 깨진 문구 복구 기준

현재 히스토리/도구 박스의 문구는 아래처럼 표시되어야 합니다.

### 박스 제목

```txt
편집 도구
```

또는

```txt
편집 도구 / 히스토리
```

### 버튼 문구

```txt
Level 1 추가
Level 1 순서변경
되돌리기 (n)
다시실행 (n)
Level 1만 보기
Level 2까지 보기
전체 펼치기
```

### 안내 문구

```txt
최근 20회의 편집 이력을 되돌리거나 다시 실행할 수 있습니다.
```

### 토스트 문구

되돌리기 실행 시:

```txt
작업이 취소되었습니다.
```

다시실행 실행 시:

```txt
작업이 다시 실행되었습니다.
```

메뉴 이동 시:

```txt
메뉴가 이동되었습니다.
```

메뉴 추가 시:

```txt
메뉴가 추가되었습니다.
```

메뉴 삭제 시:

```txt
메뉴가 삭제되었습니다.
```

## 3. 한글 깨짐 방지 요청

소스 파일 저장 인코딩은 반드시 UTF-8로 유지해 주세요.

다음 사항을 확인해 주세요.

1. 소스 파일이 UTF-8로 저장되어 있는지 확인
2. 깨진 문자열을 그대로 재사용하지 말고 정상 한글로 다시 작성
3. 문자열 변환 함수, decode/encode 처리, escape/unescape 처리가 있다면 제거 또는 수정
4. JSON 저장/불러오기 시 한글이 깨지지 않도록 처리
5. 브라우저 렌더링 시 한글 폰트가 정상 적용되도록 확인

HTML 문서 또는 Next.js 앱에서 직접 관리하는 경우, 기본적으로 UTF-8이 유지되어야 합니다.

만약 별도 HTML head를 관리하는 구조라면 아래 설정도 확인해 주세요.

```html
<meta charset="UTF-8" />
```

Next.js App Router 구조라면 파일 자체 인코딩이 UTF-8인지 특히 확인해 주세요.

# 4. 히스토리 카운트가 비정상적으로 증가하는 문제

현재 되돌리기/다시실행 히스토리 카운트가 정상적으로 1회씩 증가하지 않습니다.

예를 들어:

1. 되돌리기 5회 가능 상태
2. 되돌리기 버튼을 누름
3. 다시실행이 1회 증가해야 함
4. 그런데 실제로는 다시실행이 10회처럼 과도하게 증가함

또는 작업 한 번을 했는데 되돌리기 카운트가 2배, 3배씩 증가하는 것으로 보입니다.

즉, 히스토리가 작업 단위로 1개씩 저장되지 않고 중복 저장되고 있습니다.

---

## 5. 히스토리 중복 증가의 추정 원인

다음 중 하나 또는 여러 개가 원인일 가능성이 큽니다.

### 원인 A. 상태 변경 함수 안에서 다시 히스토리를 저장함

예를 들어 `handleUndo()` 안에서 `setMenuState()`를 호출했는데,
그 `setMenuState()`가 다시 `commitMenuChange()` 또는 히스토리 저장 로직을 타는 경우입니다.

되돌리기/다시실행은 히스토리를 새로 저장하는 작업이 아니라,
기존 히스토리 스택에서 상태를 꺼내 적용하는 작업이어야 합니다.

---

### 원인 B. useEffect에서 menuState 변경을 감지해 히스토리를 저장함

아래와 같은 구조가 있으면 문제가 생길 수 있습니다.

```js
useEffect(() => {
  saveHistory(menuState);
}, [menuState]);
```

이 방식은 위험합니다.

왜냐하면 메뉴 변경뿐 아니라 다음 상황에서도 히스토리가 계속 저장될 수 있기 때문입니다.

* undo 실행으로 menuState 변경
* redo 실행으로 menuState 변경
* 비주얼 모드 종료로 menuState 동기화
* 서버 저장 후 menuState 재세팅
* 렌더링 중 데이터 normalize
* 펼침/접힘 같은 UI 상태 반영

히스토리는 `menuState가 바뀔 때마다 자동 저장`하면 안 됩니다.

히스토리는 반드시 사용자의 실제 편집 액션에서만 저장되어야 합니다.

---

### 원인 C. 리스트 모드와 비주얼 모드가 각각 히스토리를 저장함

리스트 모드에서 한 번 저장하고,
비주얼 모드 동기화 과정에서 또 한 번 저장하면
작업 1회가 히스토리 2회 이상으로 기록될 수 있습니다.

---

### 원인 D. React Strict Mode로 개발 환경에서 함수가 두 번 실행됨

Next.js 개발 환경에서는 React Strict Mode 때문에 일부 로직이 두 번 실행될 수 있습니다.

하지만 사용자의 버튼 클릭 핸들러 자체가 기하급수적으로 증가하면 단순 Strict Mode 문제가 아니라,
히스토리 저장 로직이 여러 경로에서 중복 호출되고 있을 가능성이 높습니다.

---

# 6. 히스토리 관리 방식 수정 요청

히스토리 저장은 반드시 하나의 공통 함수에서만 처리해 주세요.

예시:

```js
const HISTORY_LIMIT = 20;

const [menuState, setMenuState] = useState(initialMenuState);
const [undoStack, setUndoStack] = useState([]);
const [redoStack, setRedoStack] = useState([]);
```

사용자가 실제로 메뉴를 수정할 때만 아래 함수를 호출합니다.

```js
function commitMenuChange(nextMenuState) {
  setUndoStack((prev) => {
    const next = [...prev, menuState];
    return next.slice(-HISTORY_LIMIT);
  });

  setRedoStack([]);
  setMenuState(nextMenuState);
}
```

중요한 점:

```txt
commitMenuChange는 메뉴 추가/삭제/이동/이름변경 같은 실제 편집 액션에서만 호출
undo/redo 내부에서는 호출 금지
useEffect에서 자동 호출 금지
렌더링/정규화/모드 전환에서 호출 금지
```

# 7. 되돌리기 로직 수정 기준

되돌리기는 다음 방식으로 동작해야 합니다.

```js
function handleUndo() {
  if (undoStack.length === 0) return;

  const previousState = undoStack[undoStack.length - 1];
  const nextUndoStack = undoStack.slice(0, -1);

  setRedoStack((prev) => {
    const next = [...prev, menuState];
    return next.slice(-HISTORY_LIMIT);
  });

  setUndoStack(nextUndoStack);
  setMenuState(previousState);

  showToast("작업이 취소되었습니다.");
}
```

주의:

```txt
handleUndo 안에서 commitMenuChange를 호출하면 안 됩니다.
handleUndo 안에서 saveHistory를 호출하면 안 됩니다.
handleUndo로 setMenuState가 실행되더라도 useEffect가 다시 history를 저장하면 안 됩니다.
```

---

# 8. 다시실행 로직 수정 기준

다시실행은 다음 방식으로 동작해야 합니다.

```js
function handleRedo() {
  if (redoStack.length === 0) return;

  const nextState = redoStack[redoStack.length - 1];
  const nextRedoStack = redoStack.slice(0, -1);

  setUndoStack((prev) => {
    const next = [...prev, menuState];
    return next.slice(-HISTORY_LIMIT);
  });

  setRedoStack(nextRedoStack);
  setMenuState(nextState);

  showToast("작업이 다시 실행되었습니다.");
}
```

주의:

```txt
handleRedo 안에서도 commitMenuChange를 호출하면 안 됩니다.
redo 실행 시 redoStack은 1개 줄고 undoStack은 1개 늘어야 합니다.
redo 한 번 실행했는데 undoStack이 2개 이상 늘어나면 안 됩니다.
```

# 10. 히스토리에 저장하면 안 되는 상태

다음 상태 변경은 히스토리에 저장하지 말아 주세요.

```txt

Level 2까지 보기
```

# 11. 리스트 모드와 비주얼 모드의 히스토리 통합

리스트 모드와 비주얼 모드 모두 같은 `commitMenuChange()`를 사용해야 합니다.

나쁜 구조:

```txt
리스트 모드용 saveHistory()
비주얼 모드용 saveVisualHistory()
X로 나갈 때 syncVisualChanges()
```

좋은 구조:

```txt
리스트 모드 메뉴 변경 → commitMenuChange(nextState)
비주얼 모드 메뉴 변경 → commitMenuChange(nextState)
X로 나가기 → 단순히 모드만 닫음, 히스토리 추가 저장 안 함
```

비주얼 모드에서 여러 번 이동했다면, 이동할 때마다 1회씩 히스토리에 저장되어야 합니다.

단, 비주얼 모드 종료 시점에 또 한 번 저장하면 안 됩니다.

# 12. 중복 저장 방지용 가드 제안

필요하다면 히스토리 저장 전에 이전 상태와 다음 상태가 실제로 다른지 비교해 주세요.

```js
function commitMenuChange(nextMenuState) {
  if (isEqual(menuState, nextMenuState)) {
    return;
  }

  setUndoStack((prev) => {
    const next = [...prev, menuState];
    return next.slice(-HISTORY_LIMIT);
  });

  setRedoStack([]);
  setMenuState(nextMenuState);
}
```

또는 JSON 기반 비교를 사용할 수 있습니다.

```js
function isSameMenuState(a, b) {
  return JSON.stringify(a) === JSON.stringify(b);
}
```

단, 메뉴 데이터가 큰 경우에는 성능을 고려해 메뉴 구조에 필요한 필드만 비교해 주세요.

# 13. 함수형 setState 사용 시 주의

`setUndoStack` 내부에서 `menuState`를 참조하면 stale state 문제가 생길 수 있습니다.

가능하면 `commitMenuChange` 호출 시점의 현재 메뉴 상태를 명확히 캡처해 주세요.

예시:

```js
function commitMenuChange(nextMenuState) {
  setHistory((prev) => {
    const nextUndo = [...prev.undoStack, menuState].slice(-HISTORY_LIMIT);

    return {
      undoStack: nextUndo,
      redoStack: [],
    };
  });

  setMenuState(nextMenuState);
}
```

또는 `menuStateRef`를 사용할 수 있습니다.

```js
const menuStateRef = useRef(menuState);

useEffect(() => {
  menuStateRef.current = menuState;
}, [menuState]);

function commitMenuChange(nextMenuState) {
  const currentState = menuStateRef.current;

  if (isSameMenuState(currentState, nextMenuState)) return;

  setUndoStack((prev) => [...prev, currentState].slice(-HISTORY_LIMIT));
  setRedoStack([]);
  setMenuState(nextMenuState);
}
```

# 핵심 요약

현재 수정 이후 다음 문제가 발생했습니다.

1. 한글 문구가 깨져 표시됨
2. 히스토리 카운트가 작업 1회당 1씩 증가하지 않고 중복/기하급수적으로 증가함

수정 방향은 다음과 같습니다.

1. 깨진 한글 문구를 모두 정상 한글로 복구
