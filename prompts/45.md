# 코드별 임시저장 기능 구현 요청

## 작업 목표

현재 메뉴 배치 테스트에서 사용자가 편집 중인 메뉴 상태를 인증코드별로 임시저장할 수 있게 하고 싶습니다.

예를 들어 인증코드가 아래처럼 있을 때:

```txt
JJST001
JJST002
````

각 코드는 서로 다른 임시저장 데이터를 가져야 합니다.

```txt
JJST001로 접속한 사용자 → JJST001의 임시저장 데이터 불러오기
JJST002로 접속한 사용자 → JJST002의 임시저장 데이터 불러오기
```

---

## 기본 방향

임시저장은 Supabase에 저장하는 방식으로 구현해 주세요.

브라우저 localStorage만 사용하는 방식은 피하고,
사용자가 다른 브라우저나 다른 기기에서 같은 인증코드로 접속해도 이어서 작업할 수 있도록 Supabase에 저장하는 것이 좋겠습니다.

---

## 수정 범위

이번 작업은 아래 기능 추가만 대상으로 합니다.

```txt
1. 코드별 임시저장 테이블 추가
2. 명시적 임시저장 버튼 추가
3. 30초 자동저장 추가
4. 로그인/입장 시 해당 코드의 임시저장 데이터 불러오기
5. 최종 제출 시 기존 제출 로직은 유지
```

---

## 수정하지 말아야 할 항목

현재 잘 동작하는 기능은 건드리지 말아 주세요.

```txt
리스트모드 드래그앤드랍
비주얼모드 드래그앤드랍
메뉴 이동/정렬 로직
되돌리기/다시실행 기본 구조
초기화 로직
최종 제출 화면
최종 제출 저장 로직
intention 저장 로직
폰트 설정
프로필 이미지
desc 툴팁
Vercel 배포 설정
인증코드 검증 방식
```

이번 작업은 **임시저장 기능 추가**가 목적입니다.

# 1. Supabase 테이블 추가 (drafts테이블 추가했음)

기존 `submissions` 테이블은 최종 제출용으로 유지하고,
임시저장 전용 테이블을 새로 추가해 주세요.

권장 테이블명:

```txt
drafts
```

또는 프로젝트 네이밍을 명확히 하고 싶다면:

```txt
menu_drafts
```

여기서는 `drafts` 기준으로 설명합니다.

---

## drafts 테이블 역할

`drafts` 테이블은 인증코드별 현재 편집 중인 메뉴 상태를 저장합니다.

최종 제출 데이터는 기존 `submissions` 테이블에 저장하고,
임시저장 데이터는 `drafts` 테이블에 저장합니다.

```txt
submissions = 최종 제출본
drafts = 작업 중 임시저장본
```

---

## Supabase SQL 예시

Supabase SQL Editor에서 아래 테이블을 추가해 주세요.

```sql
create table if not exists drafts (
  id uuid primary key default gen_random_uuid(),

  auth_code text not null unique,

  menu_data jsonb not null,
  change_log jsonb,
  undo_stack jsonb,
  redo_stack jsonb,

  intention text,

  updated_at timestamptz not null default now(),
  created_at timestamptz not null default now()
);
```

---

## 컬럼 설명

| 컬럼           | 설명                     |
| ------------ | ---------------------- |
| `id`         | 고유 ID                  |
| `auth_code`  | 인증코드, 예: `JJST001`     |
| `menu_data`  | 현재 편집 중인 메뉴 전체 구조      |
| `change_log` | 현재까지의 변경 로그            |
| `undo_stack` | 되돌리기 히스토리 저장이 필요하면 사용  |
| `redo_stack` | 다시실행 히스토리 저장이 필요하면 사용  |
| `intention`  | 최종 제출 전 작성 중인 메뉴 구성 의도 |
| `updated_at` | 마지막 임시저장 시각            |
| `created_at` | 최초 생성 시각               |

---

## unique 조건

`auth_code`는 반드시 unique로 관리해 주세요.

```sql
auth_code text not null unique
```

그래야 같은 코드로 임시저장할 때 새 row가 계속 생기지 않고,
기존 row를 업데이트할 수 있습니다.

```txt
JJST001 → drafts row 1개
JJST002 → drafts row 1개
```

로그인/입장 시 사용자가 입력한 인증코드를 기준으로 임시저장 데이터를 조회합니다.

# 3. 입장 시 임시저장 데이터 불러오기

사용자가 인증코드를 입력하고 입장할 때,
해당 인증코드의 임시저장 데이터가 있는지 확인해 주세요.

## 흐름

```txt
인증코드 입력
→ 인증 성공
→ drafts 테이블에서 auth_code로 조회
→ 임시저장 데이터가 있으면 해당 menu_data로 시작
→ 없으면 기본 initialMenu로 시작
```

---

## 예시 로직

```js
async function loadDraftByAuthCode(authCode) {
  const res = await fetch(`/api/drafts?authCode=${encodeURIComponent(authCode)}`);
  const result = await res.json();

  if (!result?.draft) {
    return null;
  }

  return result.draft;
}
```

입장 처리 예시:

```js
async function handleLoginSuccess(authCode) {
  const draft = await loadDraftByAuthCode(authCode);

  if (draft?.menu_data) {
    setMenuState(draft.menu_data);
    setChangeLog(draft.change_log || []);
    setUndoStack(draft.undo_stack || []);
    setRedoStack(draft.redo_stack || []);
    setIntention(draft.intention || "");
  } else {
    setMenuState(initialMenu);
    setChangeLog([]);
    setUndoStack([]);
    setRedoStack([]);
  }

  setIsAuthenticated(true);
}
```

# 4. 임시저장 버튼 추가

상단 편집 바에 명시적 임시저장 버튼을 추가해 주세요.

현재 상단에 이런 버튼들이 있습니다.

```txt
편집 완료
리스트 편집
화면에서 편집
초기화
미리보기
최종 제출
```

여기에 `임시저장` 버튼을 추가해 주세요.

권장 위치:

```txt
[편집 완료] [임시저장] [리스트 편집] [화면에서 편집] [초기화] [미리보기] [최종 제출]
```

또는 우측에 배치해도 됩니다.

```txt
[미리보기] [임시저장] [최종 제출]
```

---

## 임시저장 버튼 동작

사용자가 `임시저장` 버튼을 누르면 현재 상태를 Supabase `drafts` 테이블에 저장합니다.

저장 대상:

```txt
auth_code
menu_data
change_log
undo_stack
redo_stack
intention
updated_at
```

저장 성공 시 토스트:

```txt
임시저장되었습니다.
```

저장 실패 시 토스트:

```txt
임시저장 중 오류가 발생했습니다.
```

# 5. 자동저장 추가

자동저장은 30초에 한 번 실행해 주세요.

```txt
자동저장 주기: 30초
```

단, 아무 변경이 없을 때는 저장하지 않아야 합니다.

---

## 자동저장 조건

자동저장은 아래 조건을 만족할 때만 실행해 주세요.

```txt
1. 로그인/인증 완료 상태
2. authCode가 있음
3. 메뉴 상태가 이전 저장 이후 변경됨
4. 최종 제출 완료 상태가 아님
5. 현재 저장 중이 아님
```

---

## dirty 상태 관리

변경이 발생하면 `isDirty`를 `true`로 설정해 주세요.

예시:

```js
const [isDirty, setIsDirty] = useState(false);
```

메뉴 변경이 성공했을 때:

```js
commitMenuChange(nextMenuState);
setIsDirty(true);
```

메뉴 이름 변경, 추가, 삭제 등 실제 구조 변경이 있을 때도:

```js
setIsDirty(true);
```

임시저장 성공 시:

```js
setIsDirty(false);
```

---

## 자동저장 예시

```js
useEffect(() => {
  if (!isAuthenticated || !authCode) return;
  if (submitStep === "submitted") return;

  const interval = setInterval(() => {
    if (!isDirty) return;
    if (isSavingDraft) return;

    saveDraft({ silent: true });
  }, 30000);

  return () => clearInterval(interval);
}, [
  isAuthenticated,
  authCode,
  isDirty,
  isSavingDraft,
  submitStep,
  menuState,
  changeLog,
  undoStack,
  redoStack,
  intention,
]);
```

자동저장 성공 시에는 너무 큰 토스트를 띄우지 않아도 됩니다.

권장:

```txt
마지막 저장: 오후 3:21
```

처럼 작게 표시하거나, 조용히 저장해도 됩니다.

# 7. drafts 저장 API 추가

클라이언트에서 Supabase secret key를 직접 사용하지 말고,
기존 구조처럼 API Route를 사용해 주세요.

권장 API:

```txt
GET /api/drafts?authCode=JJST001
POST /api/drafts
```

---

## GET /api/drafts

인증코드 기준으로 임시저장 데이터를 조회합니다.

요청:

```txt
GET /api/drafts?authCode=JJST001
```

응답 예시:

```js
{
  draft: {
    auth_code: "JJST001",
    menu_data: [...],
    change_log: [...],
    undo_stack: [...],
    redo_stack: [...],
    intention: "",
    updated_at: "2026-05-27T..."
  }
}
```

임시저장이 없으면:

```js
{
  draft: null
}
```

---

## POST /api/drafts

현재 상태를 임시저장합니다.

요청 body:

```js
{
  authCode: "JJST001",
  menuData: currentMenuData,
  changeLog: currentChangeLog,
  undoStack: undoStack,
  redoStack: redoStack,
  intention: intentionText
}
```

서버에서는 Supabase에 upsert합니다.

```js
await supabase
  .from("drafts")
  .upsert(
    {
      auth_code: authCode,
      menu_data: menuData,
      change_log: changeLog,
      undo_stack: undoStack,
      redo_stack: redoStack,
      intention: intention || "",
      updated_at: new Date().toISOString(),
    },
    {
      onConflict: "auth_code",
    }
  );
```

# 8. 최종 제출과 임시저장 관계

최종 제출은 기존 `submissions` 테이블 저장 로직을 유지합니다.

단, 최종 제출이 성공하면 해당 코드의 임시저장을 삭제하거나 유지할지 결정해야 합니다.

권장 방식:

```txt
최종 제출 성공 후 drafts 데이터 삭제
```

이유:

```txt
이미 최종 제출이 완료되었으므로 임시저장본이 남아 있으면 다음 접속 시 다시 편집 중 상태로 오해할 수 있음
```

---

## 최종 제출 성공 후 drafts 삭제

최종 제출 성공 시:

```js
await fetch(`/api/drafts?authCode=${authCode}`, {
  method: "DELETE",
});
```

또는 서버 저장 함수 내부에서 삭제해도 됩니다.

삭제 성공/실패는 최종 제출 성공을 막지는 않도록 처리해 주세요.

---

## 대안: 제출 후에도 유지

만약 제출 후에도 사용자가 다시 확인할 수 있게 하고 싶다면 drafts를 유지해도 됩니다.

하지만 이 경우에는 `submitted` 상태를 따로 저장해야 합니다.

예시:

```sql
alter table drafts add column if not exists status text default 'draft';
```

초기에는 단순하게 가기 위해 **최종 제출 성공 후 drafts 삭제**를 권장합니다.

# 9. 자동저장과 최종 제출 충돌 방지

최종 제출 화면이나 제출 중에는 자동저장을 멈춰 주세요.

```txt
submitStep === "submitForm" → 자동저장 유지 가능
submitStep === "submitted" → 자동저장 중지
isSubmitting === true → 자동저장 중지
```

의견 textarea도 임시저장할 예정이면 `submitForm` 상태에서는 자동저장을 유지해도 됩니다.

하지만 제출 중에는 중복 요청이 생기지 않도록 막아 주세요.

# 10. 임시저장에 포함할 데이터

임시저장에는 현재 작업을 복원하는 데 필요한 값을 저장합니다.

필수:

```txt
auth_code
menu_data
change_log
intention
updated_at
```

선택:

```txt
undo_stack
redo_stack
```

권장:

```txt
undo_stack, redo_stack도 저장
```

이유:

```txt
다시 접속했을 때 되돌리기/다시실행 상태까지 유지할 수 있음
```

다만 이 부분이 복잡하거나 데이터가 너무 크면 초기에는 제외해도 됩니다.

초기 구현 최소 버전:

```txt
auth_code
menu_data
change_log
intention
updated_at
```

# 11. 30초 자동저장 적절성

자동저장은 30초에 한 번이면 적절합니다.

너무 짧게 하면 Supabase 요청이 많아지고,
너무 길게 하면 사용자가 작업을 잃을 수 있습니다.

권장:

```txt
자동저장 주기: 30초
명시적 임시저장 버튼: 즉시 저장
```

추가로 창을 닫기 전에는 저장을 시도할 수 있지만,
복잡하면 이번 작업에서는 30초 자동저장과 명시적 저장만 구현해도 됩니다.

# 12. 동일 코드 중복 접속 주의

같은 인증코드로 두 브라우저에서 동시에 접속하면,
나중에 저장한 쪽이 최신 임시저장을 덮어쓰게 됩니다.

초기에는 이 동작을 허용해도 됩니다.

```txt
동일 auth_code의 drafts row는 항상 최신 저장본 1개만 유지
```

추후 필요하면 충돌 방지 기능을 추가할 수 있습니다.

```

로 충분합니다.

# 15. 구현 시 주의사항

## 히스토리 중복 저장 금지

임시저장은 서버 저장일 뿐, 메뉴 편집 히스토리에 추가되면 안 됩니다.

```txt
임시저장 버튼 클릭 → undo/redo 변화 없음
자동저장 → undo/redo 변화 없음
draft load → undo/redo를 불필요하게 추가하지 않음
```

---

## draft load 시 commitMenuChange 사용 금지

입장 시 임시저장 데이터를 불러와서 menuState에 적용할 때는 `commitMenuChange()`를 사용하지 말아 주세요.

```txt
임시저장 불러오기 = 초기 상태 세팅
메뉴 편집 작업이 아님
```

따라서 직접 상태를 세팅해 주세요.

```js
setMenuState(draft.menu_data);
setChangeLog(draft.change_log || []);
```

if (isSavingDraft) return;
```

```js
if (!isDirty) return;
```

# 17. 최종 기대 동작

| 상황             | 기대 결과                         |
| -------------- | ----------------------------- |
| JJST001로 입장    | JJST001 draft 조회              |
| JJST002로 입장    | JJST002 draft 조회              |
| draft 없음       | 기본 메뉴로 시작                     |
| draft 있음       | 저장된 메뉴 상태로 복원                 |
| 임시저장 클릭        | 즉시 Supabase drafts에 저장        |
| 메뉴 변경 후 30초 경과 | 자동저장                          |
| 변경 없음          | 자동저장 안 함                      |
| 임시저장 성공        | 마지막 저장 시각 표시                  |
| 최종 제출 성공       | submissions 저장 후 drafts 삭제 권장 |
| 재접속            | 코드별 저장 상태에 따라 복원              |
| undo/redo      | 임시저장 때문에 카운트 변화 없음            |

# 18. 최종 요약

코드별 임시저장 기능을 Supabase 기반으로 구현해 주세요.

기본 인증코드는 우선 아래 형식을 사용합니다.

```txt
JJST001
JJST002
```

Supabase에는 최종 제출용 `submissions`와 별도로
임시저장용 `drafts` 테이블을 추가합니다.

```txt
submissions = 최종 제출
drafts = 코드별 임시저장
```

기능 요구사항:

```txt
1. auth_code 기준으로 draft 1개 유지
2. 명시적 임시저장 버튼 추가
3. 자동저장은 30초에 한 번
4. 변경이 있을 때만 자동저장
5. 입장 시 해당 auth_code의 draft 불러오기
6. 최종 제출 성공 시 draft 삭제 권장
7. 기존 편집/히스토리/제출 로직은 변경하지 않기
```
