# Drag Preview 위치와 실제 Drop 위치 불일치 공통 점검 요청

## 현재 문제

Visual Editor Mode에서 드래그 중 표시되는 preview 위치와  
마우스를 놓은 뒤 실제로 삽입되는 위치가 다릅니다.

예시:

- 드래그 중에는 특정 메뉴 위치에 들어갈 것처럼 표시됨
- 실제 drop 후에는 예상보다 한 칸 왼쪽 또는 다른 위치에 삽입됨

추가로, 이 문제가 Visual Editor Mode뿐 아니라  
List Mode에서도 동일하게 발생할 가능성이 있습니다.

따라서 이번에는 Visual Mode만 단정하지 말고,  
**List Mode와 Visual Mode 모두에서 preview 위치와 실제 drop 결과가 일치하는지 점검**해주세요.

---

# 중요 전제

List Mode는 현재 전체 기능이 비교적 정상 동작하고 있으므로,  
기존 기능을 깨지 않도록 주의해주세요.

이번 수정의 핵심은 기능 변경이 아니라 아래 문제를 보정하는 것입니다.

```txt
드래그 중 보이는 위치 = 실제 drop 후 적용 위치
````

---

# 수정 범위

점검 대상:

* List Mode drag preview 계산
* List Mode 실제 drop insertIndex 계산
* Visual Mode drag preview 계산
* Visual Mode 실제 drop insertIndex 계산
* 공통 drag/drop utility
* reorder apply 로직
* moving item 제거 후 targetIndex 계산 로직

단, 아래 기능은 유지해야 합니다.

```txt
List Mode 기존 레벨 제한 유지
Visual Mode 기존 레벨 제한 유지
Level 1/2/3 구조 유지
Level 4 생성 금지
```

---

# 현재 의심되는 원인

현재 preview 계산과 실제 drop 적용 계산이 서로 다른 기준을 쓰고 있을 가능성이 큽니다.

가능한 원인:

```txt
1. hover preview는 마우스 좌표 기준으로 계산함
2. 실제 drop은 배열 index 기준으로 다시 계산함
3. dragItem을 제거하기 전 targetIndex를 계산해서 index가 한 칸 밀림
4. preview의 position(before/after)과 실제 drop의 position이 다름
5. targetItem은 같지만 insertIndex 계산 방식이 다름
6. List Mode와 Visual Mode가 서로 다른 drop intent 계산 함수를 사용함
7. CSS grid/flex 시각 순서와 데이터 배열 순서가 다름
8. children이 있는 항목을 그룹으로 이동하지 않고 단일 item만 이동함
```

---

# 핵심 수정 방향

preview용 계산과 실제 drop용 계산을 분리하지 말고,
하나의 drop intent 객체를 만들어서 동일하게 사용해주세요.

```js
const dropIntent = {
  mode,        // 'list' | 'visual'
  dragId,
  targetId,
  dragLevel,
  targetLevel,
  dropMode,   // 'reorder' | 'move-to-parent' | 'move-as-child'
  position,   // 'before' | 'after' | 'inside'
  newParentId,
};
```

이 `dropIntent`를 아래 세 곳에서 동일하게 사용해야 합니다.

```txt
1. 드래그 중 preview 표시
2. drop 가능 여부 검증
3. 실제 drop 적용
```

---

# 잘못된 방식

아래처럼 preview와 drop 시점에 위치를 따로 계산하면 안 됩니다.

```js
// hover 시점
const previewIndex = calculatePreviewIndex(mousePosition);

// drop 시점
const insertIndex = calculateInsertIndexAgain(nodes, targetItem);
```

이렇게 하면 preview와 실제 결과가 달라질 수 있습니다.

---

# 올바른 방식

hover 시점에 계산한 drop intent를 저장하고,
drop 시점에는 그 intent를 그대로 사용해주세요.

```js
const intent = getDropIntent({
  mode,
  dragItem,
  targetItem,
  position,
  nodes,
});

setDropPreview(intent);

// drop 시
applyDropByIntent(intent, nodes);
```

---

# insertIndex 계산 주의사항

실제 drop 적용 시에는 반드시
**moving item 또는 moving group을 먼저 제거한 뒤 targetIndex를 다시 계산**해야 합니다.

잘못된 방식:

```js
const targetIndex = nodes.findIndex(node => node.id === targetItem.id);
const remainingNodes = nodes.filter(node => node.id !== dragItem.id);
```

이 방식은 dragItem이 targetItem보다 앞에 있던 경우
index가 한 칸 어긋날 수 있습니다.

올바른 방식:

```js
const movingIds = new Set(movingGroup.map(node => node.id));

const remainingNodes = nodes.filter(node => !movingIds.has(node.id));

const targetIndex = remainingNodes.findIndex(
  node => node.id === intent.targetId
);

const insertIndex =
  intent.position === 'after'
    ? targetIndex + 1
    : targetIndex;
```

# List Mode도 동일하게 점검

List Mode에서도 아래 상황을 테스트해주세요.

| 테스트                                | 기대 결과        |
| ---------------------------------- | ------------ |
| Level 1을 다른 Level 1 앞에 preview     | 실제로도 그 앞에 삽입 |
| Level 1을 다른 Level 1 뒤에 preview     | 실제로도 그 뒤에 삽입 |
| Level 2를 같은 부모 안에서 preview         | 실제 위치와 일치    |
| Level 2를 다른 Level 1 아래로 이동 preview | 실제 위치와 일치    |
| Level 3을 같은 부모 안에서 preview         | 실제 위치와 일치    |
| Level 3을 다른 Level 2 아래로 이동 preview | 실제 위치와 일치    |

List Mode의 기존 제한사항은 유지하되,
preview와 실제 적용 위치가 다른 경우만 보정해주세요.

---

# Visual Mode도 동일하게 점검

Visual Mode에서도 아래 상황을 테스트해주세요.

| 테스트                            | 기대 결과                |
| ------------------------------ | -------------------- |
| Level 2를 다른 Level 2 앞에 preview | 실제로도 그 앞에 삽입         |
| Level 2를 다른 Level 2 뒤에 preview | 실제로도 그 뒤에 삽입         |
| Level 2를 다른 Level 1 아래로 이동     | preview 위치와 실제 위치 일치 |
| Level 3을 다른 Level 3 앞/뒤로 이동    | preview 위치와 실제 위치 일치 |
| Level 3을 다른 Level 2 아래로 이동     | preview 위치와 실제 위치 일치 |

---

# parentId 처리 기준

preview와 drop 결과가 같더라도 parentId가 틀어지면 안 됩니다.

## Level 1

```txt
parentId 없음
하위 Level 2/3 함께 이동
```

## Level 2

```txt
Level 2 reorder 시:
newParentId = target Level 2가 속한 Level 1 id

Level 2를 Level 1 아래로 이동 시:
newParentId = target Level 1 id
```

## Level 3

```txt
Level 3 reorder 시:
newParentId = target Level 3이 속한 Level 2 id

Level 3을 Level 2 아래로 이동 시:
newParentId = target Level 2 id
```

---

# children 그룹 이동 처리

Level 1과 Level 2는 하위 메뉴가 있으므로,
단일 item만 이동하면 구조가 깨질 수 있습니다.

따라서 아래처럼 그룹 단위 이동이 필요합니다.

```txt
Level 1 이동:
Level 1 + 하위 Level 2 + 하위 Level 3 함께 이동

Level 2 이동:
Level 2 + 하위 Level 3 함께 이동

Level 3 이동:
Level 3 단독 이동
```

# 디버깅 로그 추가 요청

preview 계산 시점과 drop 적용 시점에 같은 값을 찍어주세요.

```js
console.log('Preview drop intent:', {
  mode,
  dragTitle: dragItem.title,
  targetTitle: targetItem.title,
  dropMode: intent.dropMode,
  position: intent.position,
  newParentId: intent.newParentId,
  targetId: intent.targetId,
});
```

```js
console.log('Apply drop intent:', {
  mode,
  dragId: intent.dragId,
  targetId: intent.targetId,
  dropMode: intent.dropMode,
  position: intent.position,
  newParentId: intent.newParentId,
});
```

두 로그가 다르면 preview와 실제 drop 계산이 분리된 것입니다.

# 완료 기준

| 항목                     | 기대 결과                |
| ---------------------- | -------------------- |
| List Mode preview 위치   | 실제 drop 위치와 일치       |
| Visual Mode preview 위치 | 실제 drop 위치와 일치       |
| before preview         | 실제로도 before 삽입       |
| after preview          | 실제로도 after 삽입        |
| inside preview         | 실제로도 해당 parent 아래 삽입 |
| 이동 후 parentId          | 올바른 부모 id 유지         |
| Level 1 이동             | 하위 Level 2/3 함께 이동   |
| Level 2 이동             | 하위 Level 3 함께 이동     |
| Level 3 이동             | 단독 이동                |
| Level 4 생성             | 절대 없음                |
| 기존 레벨 제한               | 유지                   |

# 최종 요약

이번 문제는 Visual Mode만의 문제가 아니라
List Mode에서도 발생할 수 있는 **공통 drag/drop index 계산 문제**로 보입니다.

따라서 두 모드 모두 아래 원칙을 적용해주세요.

```txt
preview에서 계산한 targetId / position / newParentId를
drop 시점에도 그대로 사용한다.
```

그리고 실제 배열 적용 시에는 반드시:

```txt
movingGroup 제거 후 targetIndex를 다시 계산한다.
```

List Mode의 기존 기능을 새로 바꾸는 것이 아니라,
preview와 실제 drop 결과가 불일치하는 문제만 보정해주세요.

```
