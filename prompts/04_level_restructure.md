# 메뉴 편집기 구조 개선 요구사항 정리

## 0. 목적

현재 메뉴 편집기는 사용자가 직접 메뉴의 Level 1, Level 2, Level 3 구조를 재배치하고, 이름 변경, 삭제, 추가, 이동, 통합, 임시저장, 최종 제출까지 수행하는 도구이다.

기존 문제는 메뉴가 화면상으로는 트리처럼 보이지만, 실제 편집 로직에서 flat list와 level 값을 기준으로 부모-자식 관계를 추론하면서 발생했다.

이 문서는 메뉴 편집기에서 반드시 지켜야 할 데이터 구조, 드래그 규칙, 명시적 버튼 규칙, 삭제/통합 규칙, 로그/되돌리기/검증 규칙을 정리한다.

---

## 1. 데이터 구조 원칙

### 1.1 기본 메뉴 데이터

기본 메뉴 데이터는 `menus.js`에서 관리한다.

기본 데이터에는 다음 정보만 필요하다.

```txt
title
children
````

단, 편집기 내부에서는 각 메뉴를 안정적으로 식별하기 위해 `id`를 생성해서 사용한다.

### 1.2 원본 데이터는 tree 구조로 관리

메뉴 원본은 반드시 tree 구조로 관리한다.

```js
[
  {
    id: "menu-0001",
    title: "학사정보",
    children: [
      {
        id: "menu-0002",
        title: "수강관리",
        children: [
          {
            id: "menu-0003",
            title: "수강신청",
            children: []
          }
        ]
      }
    ]
  }
]
```

### 1.3 level 값은 저장하지 않음

`level`은 저장 데이터의 원본값으로 사용하지 않는다.

Level은 children 깊이로 판단한다.

```txt
menus 배열의 항목 = Level 1
Level 1의 children = Level 2
Level 2의 children = Level 3
```

### 1.4 flat list는 렌더링용으로만 사용

리스트 모드 화면에 보여줄 때만 tree를 flat list로 변환한다.

```js
function flattenTree(nodes, level = 1, parentId = null) {
  return nodes.flatMap((node) => [
    {
      id: node.id,
      title: node.title,
      level,
      parentId,
      hasChildren: node.children.length > 0
    },
    ...flattenTree(node.children, level + 1, node.id)
  ]);
}
```

금지 사항:

```txt
화면 순서 + level 숫자만 보고 부모-자식 관계를 다시 추론하지 않는다.
```

모든 이동, 삭제, 추가, 통합, 레벨 변경은 반드시 tree 기준으로 처리한다.

## 2. 메뉴 상태 관리

### 2.1 기본 상태와 사용자 편집 상태 분리

메뉴 상태는 항상 별도로 관리되어야 한다.

```txt
기본 메뉴 상태: 앱이 알고 있는 기본 menus.js 또는 서버 기본값
사용자 편집 상태: 사용자가 편집 중인 menuTree
임시저장 상태: 서버 또는 로컬 저장소에 저장된 draft menuTree
최종 제출 상태: 검증을 통과한 submitted menuTree
```

### 2.2 초기화

사용자는 편집 상태를 초기화할 수 있어야 한다.

초기화 시에는 사용자 편집 상태를 버리고 기본 메뉴 상태로 되돌린다.

### 2.3 임시저장 및 이어하기

사용자는 편집 중인 메뉴를 임시저장할 수 있어야 한다.

이후 다시 접속했을 때 임시저장된 menuTree를 불러와 이어서 편집할 수 있어야 한다.

### 2.4 서버 통신

서버와 통신할 때는 flat list가 아니라 tree 구조를 저장한다.

```js
{
  menuTree,
  changeLogs,
  updatedAt,
  status: "draft" | "submitted"
}
```

## 3. 구조 제한 규칙

전체 메뉴 구조는 최대 3레벨까지만 허용한다.

```txt
Level 1
  Level 2
    Level 3
```

### 3.1 Level 1 규칙

```txt
Level 1은 항상 최상위 root에만 존재한다.
Level 1은 Level 2나 Level 3의 자식이 될 수 없다.
Level 1은 다른 Level 1의 자식이 될 수 없다.
Level 1은 Level Up / Level Down 대상이 아니다.
```

### 3.2 Level 2 규칙

```txt
Level 2는 반드시 Level 1의 직접 자식이어야 한다.
Level 2는 다른 Level 2의 자식이 될 수 없다.
Level 2는 Level 3의 자식이 될 수 없다.
Level 2는 다른 Level 1 아래로 이동할 수 있다.
Level 2를 이동할 때는 하위 Level 3까지 그룹으로 함께 이동한다.
```

### 3.3 Level 3 규칙

```txt
Level 3은 반드시 Level 2의 직접 자식이어야 한다.
Level 3은 Level 1의 직접 자식이 될 수 없다.
Level 3은 Level 3의 자식이 될 수 없다.
Level 3은 다른 Level 2 아래로 이동할 수 있다.
```

---

## 4. Level 1 관리 정책

### 4.1 Level 1 항목 내부 버튼

Level 1 메뉴 항목 내부에는 다음 버튼만 둔다.

```txt
이름변경
삭제
```

Level 1 항목 내부에 두면 안 되는 버튼:

```txt
Level Up
Level Down
들여쓰기
내어쓰기
하위로 이동
하위 메뉴 추가
```

### 4.2 Level 1 추가 및 순서변경

Level 1 추가와 Level 1 순서변경은 항목 내부가 아니라 상단 관리 영역에서 처리한다.

상단 버튼 예시:

```txt
[+ Level 1 추가] [Level 1 순서변경]
```

### 4.3 Level 1 순서변경

Level 1 순서변경 모드에 들어가면 모든 Level 1이 그룹 단위로 움직여야 한다.

Level 1 순서변경 시 실제 expanded 상태는 유지하되, 화면 연출상 모든 Level 1이 닫힌 것처럼 보여준다.

```txt
실제 expandedIds: 유지
드래그 중 표시 상태: 닫힘처럼 연출
드래그 종료 후: 기존 expandedIds 복원
```

Level 1 순서변경은 Level 1 그룹 전체를 이동하는 것이다.

```txt
Level 1 + 그 하위 Level 2 + 그 하위 Level 3 전체가 함께 이동
```

## 5. 리스트모드와 비주얼모드 상태 동기화

리스트모드와 비주얼모드는 서로 다른 화면일 뿐, 반드시 같은 `menuTree`를 사용해야 한다.

```txt
공통 원본 상태: menuTree
리스트모드: menuTree를 flatten해서 표시
비주얼모드: menuTree를 카드/박스 구조로 표시
```

금지 사항:

```txt
리스트모드용 상태와 비주얼모드용 상태를 따로 관리하지 않는다.
```

어느 모드에서 변경하든 공통 menuTree가 변경되고, 다른 모드도 이를 다시 렌더링해야 한다.

## 6. 리스트모드 드래그 정책

### 6.1 기본 방향

리스트모드 드래그는 허용한다.

단, 드롭 위치를 명확히 구분해야 한다.

### 6.2 담기 드롭존

각 메뉴의 가장 왼쪽 또는 리스트 바깥쪽에 장바구니 모양 SVG 아이콘을 둔다.

이 아이콘 영역만 “담기 드롭존”으로 사용한다.

```txt
왼쪽 장바구니 아이콘 드롭존 = 해당 메뉴 안으로 넣기 / 통합
그 외 영역 = 순서 정렬
```

드롭존 UI:

```txt
모서리가 둥근 네모 배경
장바구니 또는 담기 SVG 아이콘
드래그 중 활성화
드래그 오버 시 노란색 하이라이트
결과 문구 표시
```

예시 문구:

```txt
"학사정보" 하위로 이동
"수강관리"에 하위메뉴 통합
Level 3은 Level 1의 직접 하위로 이동할 수 없습니다.
```

### 6.3 순서 정렬

담기 아이콘 드롭존이 아닌 영역에 놓으면 모두 순서 정렬로 처리한다.

```txt
행 위
행 아래
행 사이
리스트 빈 공간
```

순서 정렬 시에는 해당 레벨 내에서만 이동한다.

### 6.4 드래그 중 그룹 축소 연출

Level 2를 드래그할 때는 하위 Level 3까지 포함한 그룹으로 움직이는 것처럼 보여야 한다.

예시:

```txt
장학관리 L2 외 하위메뉴 5개
```

드래그 대상이 Level 2면 subtree 전체가 이동 대상이다.

## 7. 드래그 이동 규칙

### 7.1 Level 1 드래그

일반 리스트모드에서 Level 1은 드래그하지 않는다.

Level 1 순서변경은 별도 상단 모드에서만 허용한다.

### 7.2 Level 2를 Level 1 드롭존에 넣는 경우

Level 2를 다른 Level 1의 담기 드롭존에 놓으면 해당 Level 1의 자식으로 이동한다.

이때 하위 Level 3도 함께 이동한다.

팝업 문구:

```txt
"{Level 2 이름}"의 하위메뉴도 함께 이동하시겠습니까?
```

버튼:

```txt
[함께 이동] [취소]
```

취소 시 아무 작업도 하지 않는다.

### 7.3 Level 2를 Level 2 드롭존에 넣는 경우

Level 2는 Level 2의 자식이 될 수 없다.

따라서 이 경우 “통합”으로 처리한다.

처리 방식:

```txt
이동하려는 Level 2 자체는 제거
그 하위 Level 3 메뉴만 대상 Level 2의 children으로 이동
```

팝업 문구:

```txt
"{이동할 Level 2 이름}"은 Level 2 메뉴이므로 "{대상 Level 2 이름}"의 하위 메뉴가 될 수 없습니다.

대신 "{이동할 Level 2 이름}" 메뉴를 제거하고,
그 하위메뉴를 "{대상 Level 2 이름}"의 하위메뉴로 통합할 수 있습니다.

진행하시겠습니까?
```

버튼:

```txt
[통합하기] [취소]
```

통합 시 원래 Level 2는 화면에 `[삭제됨]` 상태로 남기지 않는다.

대신 변경 로그에 통합 기록을 남긴다.

### 7.4 Level 3을 Level 2 드롭존에 넣는 경우

Level 3을 다른 Level 2의 담기 드롭존에 놓으면 해당 Level 2의 자식으로 이동한다.

### 7.5 금지 드롭

다음은 금지한다.

```txt
Level 1 → Level 2 하위
Level 1 → Level 3 하위
Level 2 → Level 3 하위
Level 3 → Level 1 직접 하위
Level 3 → Level 3 하위
자기 자신의 후손 하위로 이동
Level 4 이상 발생
```

## 8. 명시적 버튼 정책

### 8.1 위/아래 순서 버튼

명시적 위/아래 순서 버튼은 같은 레벨 내 순서 이동만 처리한다.

즉, 레벨을 바꾸지 않는다.

```txt
Level 2 위/아래 버튼 → 같은 Level 1 안의 다른 Level 2 사이에서만 이동
Level 3 위/아래 버튼 → 같은 Level 2 안의 다른 Level 3 사이에서만 이동
```

다른 Level 1이나 다른 Level 2로 넘어가면 안 된다.

### 8.2 Level 1 버튼

Level 1에는 명시적 Level Up / Level Down 버튼이 없다.

Level 1은 이름변경, 삭제만 가진다.

### 8.3 Level 2 Level Down

Level 2를 Level Down 하면 Level 3이 된다.

단, Level 2에 하위 Level 3이 없어야 한다.

자식이 있는 Level 2는 Level Down 금지.

금지 문구:

```txt
"{메뉴명}"에는 하위메뉴가 있어 Level 3으로 변경할 수 없습니다.
먼저 하위메뉴를 다른 Level 2로 이동하거나 삭제해 주세요.
```

#### Level 2 Down 위치 규칙

Level 2를 Level Down 할 때는 같은 Level 1 안의 형제 Level 2만 대상이 된다.

규칙:

```txt
바로 뒤에 같은 부모의 Level 2가 있으면 그 Level 2의 첫 번째 Level 3으로 이동
뒤에 Level 2가 없으면 바로 앞 같은 부모의 Level 2의 마지막 Level 3으로 이동
앞뒤에 받을 수 있는 Level 2가 없으면 Level Down 불가
```

예시:

```txt
A Level 2
B Level 2
  C Level 3
  D Level 3
```

A를 Level Down:

```txt
B Level 2
  A Level 3
  C Level 3
  D Level 3
```

뒤에 Level 2가 없는 경우:

```txt
B Level 2
  C Level 3
  D Level 3
A Level 2
```

A를 Level Down:

```txt
B Level 2
  C Level 3
  D Level 3
  A Level 3
```

### 8.4 Level 3 Level Up

Level 3을 Level Up 하면 Level 2가 된다.

위치 규칙:

```txt
현재 부모 Level 2의 바로 위쪽에 Level 2로 올라간다.
```

예시:

```txt
B Level 2
  A Level 3
  C Level 3
  D Level 3
```

A를 Level Up:

```txt
A Level 2
B Level 2
  C Level 3
  D Level 3
```

Level Up 후에는 변경된 위치로 자동 스크롤한다.

### 8.5 Level 버튼은 해당 아이템 하나에만 적용

Level Up / Level Down 버튼은 클릭한 해당 아이템 하나에만 적용한다.

금지 사항:

```txt
아래에 있는 다른 메뉴들을 자동으로 자식으로 흡수하지 않는다.
자식 전체의 level을 자동으로 함께 변경하지 않는다.
```

## 9. 삭제 정책

### 9.1 자식 없는 메뉴 삭제

자식이 없는 메뉴는 팝업 없이 즉시 삭제한다.

예시:

```txt
Level 3 삭제 → 바로 삭제
```

삭제 후에는 Undo 토스트를 제공한다.

```txt
"{메뉴명}"이 삭제되었습니다. [실행 취소]
```

### 9.2 자식 있는 메뉴 삭제

자식이 있는 메뉴는 바로 삭제하지 않고 처리 방식을 선택하게 한다.

팝업 문구:

```txt
"{메뉴명}"에는 하위메뉴가 있습니다.
하위메뉴를 어떻게 처리하시겠습니까?
```

버튼:

```txt
[함께 삭제] [이동 메뉴 선택] [취소]
```

### 9.3 함께 삭제

함께 삭제를 선택하면 해당 메뉴와 모든 하위메뉴를 삭제한다.

삭제 내용은 변경 로그에 기록한다.

### 9.4 이동 메뉴 선택

하위메뉴를 이동시킬 경우, 하위메뉴를 받을 수 있는 부모 후보 목록을 표시한다.

부모 후보는 삭제 대상과 같은 Level의 다른 메뉴로 제한한다.

정확한 의미:

```txt
Level 1 삭제 시:
- 삭제 대상의 하위 Level 2 메뉴를 받을 수 있는 다른 Level 1 메뉴만 표시한다.

Level 2 삭제 시:
- 삭제 대상의 하위 Level 3 메뉴를 받을 수 있는 다른 Level 2 메뉴만 표시한다.

Level 3 삭제 시:
- 하위메뉴가 없으므로 이동 대상 선택 없이 바로 삭제한다.
```

### 9.5 삭제 시 새 그룹 추가

삭제 팝업에서 새 그룹 추가 기능은 아직 보류한다.

구체 정책은 추후 별도로 확정한다.

## 10. 통합 정책

### 10.1 통합 발생 조건

통합은 Level 2를 다른 Level 2의 드롭존에 넣으려 할 때 발생한다.

### 10.2 통합 처리

```txt
이동하려는 Level 2는 제거
이동하려는 Level 2의 하위 Level 3들은 대상 Level 2의 children으로 이동
```

### 10.3 통합 기록

화면에는 `[삭제됨]` 그룹을 남기지 않는다.

대신 변경 로그에 반드시 기록한다.

예시 로그:

```txt
"장학관리" 메뉴가 제거되고, 하위메뉴 5개가 "수강관리"로 통합되었습니다.
```

## 11. 변경 로그 정책

### 11.1 모든 주요 변경 기록

다음 작업은 변경 로그에 기록한다.

```txt
메뉴 이동
순서 변경
Level Up
Level Down
삭제
통합
이름 변경
Level 1 추가
임시저장
초기화
```

### 11.2 로그와 Undo 연결

변경 로그는 단순 표시용이 아니라 Undo 기능과 연결한다.

권장 구조:

```js
{
  id: "log-001",
  type: "move",
  summary: '"장학관리"가 "학사정보" 아래로 이동되었습니다.',
  before: previousMenuTree,
  after: nextMenuTree,
  createdAt: "2026-05-14T13:16:00"
}
```

Undo는 가장 최근 로그의 `before` 상태로 되돌리는 방식으로 처리한다.

### 11.3 변경사항 보기

최종 제출 전 “변경사항 보기”를 제공한다.

화면 구성은 추후 확정하되, 현재 기준은 다음과 같다.

```txt
큰 틀 안에 작은 폰트의 로그 리스트 표시
10 x 10 정도의 넓은 리스트 또는 카드형 요약
비주얼모드처럼 큰 구조를 보되, 상세는 작게 표시
```

목적:

```txt
어떤 메뉴가 어디로 이동했는지
어떤 메뉴가 삭제되었는지
어떤 메뉴가 생성되었는지
어떤 메뉴가 통합되었는지
어떤 메뉴명이 변경되었는지
```

## 12. Undo 기능

### 12.1 Undo 필수

삭제, 통합, 이동, Level 변경은 실수 가능성이 있으므로 Undo 기능을 추가한다.

### 12.2 Undo 방식

가장 최근 변경 로그의 `before` 상태로 되돌린다.

### 12.3 UI

작업 직후 토스트를 표시한다.

```txt
"장학관리"가 "수강관리"로 통합되었습니다. [실행 취소]
```

## 13. 열림/닫힘 상태 정책

### 13.1 실제 상태와 연출 상태 분리

사용자가 열어둔 메뉴 상태는 유지한다.

다만 드래그나 순서변경 중에는 화면상 닫힌 것처럼 연출할 수 있다.

```txt
expandedIds = 실제 열림 상태
dragPreviewMode = 드래그 중 닫힘 연출 상태
```

### 13.2 순서 정렬 후

순서 정렬이 완료되면 비교 대상이 된 그룹은 닫힌 것처럼 보여 결과를 확인하기 쉽게 한다.

단, 실제 expanded 상태를 완전히 잃지 않도록 주의한다.

## 14. 중복 이름 경고

기존 메뉴에는 중복 이름이 있을 수 있다.

중복 이름 자체를 무조건 금지하지는 않는다.

다만 같은 부모 안에 동일한 메뉴명이 있으면 경고한다.

권장 문구:

```txt
같은 그룹 안에 동일한 메뉴명이 있습니다.
그래도 저장하시겠습니까?
```

중복 메뉴를 사용자가 직접 정리하는 것이 이 편집기의 목적이므로, 경고는 하되 강제 금지는 하지 않는다.

---

## 15. 최종 제출 전 검증

최종 제출 전에는 반드시 전체 tree 검증을 수행한다.

검증 항목:

```txt
1. Level 1은 root에만 존재하는가
2. Level 2의 부모는 반드시 Level 1인가
3. Level 3의 부모는 반드시 Level 2인가
4. Level 4 이상이 없는가
5. parent가 끊긴 orphan 메뉴가 없는가
6. 빈 title 메뉴가 없는가
7. 중복 이름 경고 대상이 있는가
8. 삭제 로그 또는 통합 로그가 남아 있는가
9. 리스트모드와 비주얼모드가 같은 menuTree를 바라보고 있는가
```

검증 실패 시 최종 제출을 막는다.

검증 경고만 있는 경우에는 사용자 확인 후 제출 가능하게 할 수 있다.

---

## 16. 최종 제출 시 삭제/통합 확인

삭제된 메뉴나 통합으로 제거된 메뉴가 있는 경우 최종 제출 시 경고한다.

문구:

```txt
삭제되거나 통합으로 제거된 메뉴가 있습니다.

※ 삭제/통합된 메뉴는 최종 제출 후 복구할 수 없습니다.
최종 상태로 제출하시겠습니까?
```

빨간 문구로 강조한다.

버튼:

```txt
[동의하고 제출] [취소]
```

## 17. 구현상 주의사항

### 17.1 flat list를 원본으로 쓰지 말 것

현재 가장 큰 문제는 flat list와 level 값을 기준으로 부모-자식 관계를 재추론하는 것이다.

반드시 tree 구조를 원본으로 두고, flat list는 렌더링용으로만 사용한다.

### 17.2 DnD ID 분리

리스트모드에서 정렬 영역과 담기 드롭존을 분리해야 한다.

권장 ID 구조:

```txt
sort:{menuId}
drop-into:{menuId}
```

예시:

```txt
sort:menu-0003
drop-into:menu-0003
```

`sort:` 영역은 순서 정렬만 처리한다.

`drop-into:` 영역은 하위 이동 또는 통합만 처리한다.

### 17.3 모든 작업은 순수 함수화

메뉴 변경 로직은 가능하면 순수 함수로 분리한다.

예시:

```txt
moveNode()
moveNodeInto()
mergeLevel2IntoLevel2()
deleteNode()
renameNode()
levelUpNode()
levelDownNode()
validateTree()
flattenTree()
```

UI 컴포넌트 안에서 직접 복잡한 tree 조작을 하지 않는다.

## 18. 최종 요약

```txt
1. 원본 메뉴는 tree 구조로 관리한다.
2. level은 저장하지 않고 children 깊이로 판단한다.
3. flat list는 렌더링용으로만 사용한다.
4. 리스트모드와 비주얼모드는 같은 menuTree를 공유한다.
5. Level 1은 이름변경/삭제만 항목 내부에서 제공한다.
6. Level 1 추가/순서변경은 상단 관리 영역에서 처리한다.
7. Level 2 이동 시 하위 Level 3까지 그룹으로 이동한다.
8. Level 2를 Level 2에 넣으려 하면 통합으로 처리한다.
9. Level 3은 다른 Level 2로 이동할 수 있다.
10. 명시적 위/아래 버튼은 같은 레벨 내 순서 이동만 처리한다.
11. 명시적 Level 버튼은 해당 아이템 하나에만 적용한다.
12. 자식 없는 메뉴는 팝업 없이 삭제한다.
13. 자식 있는 메뉴는 함께 삭제/이동 메뉴 선택/취소를 제공한다.
14. 변경 로그를 남기고 Undo와 연결한다.
15. 최종 제출 전 검증을 통과해야 한다.
16. 삭제/통합된 메뉴가 있으면 최종 제출 시 빨간 경고를 표시한다.
```
